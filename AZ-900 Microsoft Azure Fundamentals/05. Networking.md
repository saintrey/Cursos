# 05. Networking (Redes en Azure)

## Resumen del Capítulo
Networking es fundamental para la computación en nube. Este capítulo cubre cómo se comunican recursos Azure entre sí y con el internet, incluyendo redes virtuales, balanceadores de carga, VPN, y distribución de contenidos global.

## Temas Principales

### 1. Virtual Networks (VNets - Redes Virtuales)

#### a) Definición
- **Red privada virtual** que habilita recursos Azure a comunicarse
- Similar a red tradicional en data center
- Virtual: No administras hardware (es de Azure)
- Permiten comunicación: Entre servicios, con internet, con redes on-premises

#### b) Conceptos Clave

**1. Address Space (Rango de Direcciones IP)**
- Rango de direcciones IP disponibles
- Análogo a calle para direcciones físicas
- Cada recurso conectado obtiene IP única dentro del address space
- Los recursos usan estas IPs para comunicarse

**2. Subnets (Subredes)**
- Segmentan la VNet en sub-redes
- Cada subnet = Porción del address space
- **Beneficios**:
  - Mejor tracking de grupos de servicios
  - Asignación IP más eficiente
  - **Seguridad**: Puedes asegurar subredes individualmente con NSGs

**3. Propiedades Importantes**
- **Una región**: Todos recursos en VNet deben estar en MISMO parasite
- **Una suscripción**: Cada VNet pertenece a una suscripción
- Puedes conectar VNets (peering) para comunicación cross-region

#### c) Ventajas de VNets Cloud vs On-Premises
- **Escalabilidad**: Con click, agrega/quita direcciones
- **Alta Disponibilidad**: A través de VNet peering o load balancers
- **Aislamiento**: Servicios seguros con subnets y NSGs
- **Flexibility**: Rearrangements sin costo de hardware

#### d) VNet Peering
- Conecta 2+ redes virtuales en Azure
- Tráfico entre VNets usa **backbone privado Microsoft** (NO internet público)
- Misma latencia como si estuvieran en misma VNet

**Beneficios**:
- Conexión baja-latencia, alta-bandwidth
- Recursos en VNets separadas pueden comunicarse
- Cross-suscripción, cross-región posible
- Caso de uso: Múltiples VNets de una misma empresa

### 2. Load Balancer (Balanceador de Carga)

#### a) Problema Que Resuelve
```
Tengo 2+ VMs sirviendo mi app
¿A cuál VM envío el traffic?
→ Load Balancer distribuye traffic inteligentemente
```

#### b) Definición
- Distribuye incoming traffic entre backend VMs
- Basado en reglas y health probes
- Asegura distribución equitativa

#### c) Componentes
- **Inbound Flows**: Tráfico entrante (internet o VNet local)
- **Frontend**: Punto de entrada único (los usuarios ven esto)
- **Backend Pool**: VMs que reciben el tráfico
- **Rules & Health Probes**: Define cómo distribuir y verificar salud

#### d) Health Probes
- Verifica si VM está lista para recibir tráfico
- Solo envía traffic a VMs saludables
- Si VM falla, load balancer lo detecta y redirige

#### e) Casos de Uso
- Load balancing de tráfico incoming
- Traffic desde redes internas
- Port forwarding (redirige a puerto específico)
- Outbound connectivity para VMs backend

### 3. VPN Gateway

#### a) Problema: Conexión Secure a On-Premises
```
Tengo infraestructura on-premises y también en Azure
Necesito conectar securely
→ VPN Gateway conecta hybrid cloud
```

#### b) Definición
- **Virtual Network Gateway** especializado
- Tipo específico de gateway VNet
- Envía tráfico encrypted entre Azure VNet y on-premises

#### c) Componentes VPN Gateway Setup
1. **Azure VNet con VPN Gateway** (su propia IP pública)
2. **Secure Tunnel** (conexión encrypted por internet)
3. **On-Premises Network Gateway** (en tu datacenter)

#### d) Tipos de Conexiones
- **Site-to-Site**: Un on-premises con múltiples Azure
- **Multi-Site**: Múltiples on-premises con un Azure
- Encrypted data traverses public internet segura

#### e) Usar Cases
- Hybrid cloud architecture
- Migración gradual on-premises → cloud
- Datos sensibles que deben estar on-premises

### 4. Application Gateway

#### a) Diferencia con Load Balancer
```
Load Balancer:
  - Distribuye por: IP address + port
  - Layer 3/4 (Red/Transporte)

Application Gateway:
  - Distribuye por: HTTP request properties
  - Layer 7 (Aplicación)
```

#### b) Distribución Inteligente
- Basada en: **URI path** (example: /images → image servers)
- Basada en: **Host headers** (example: videos.com → video servers)
- Mismo tráfico → múltiples backend pools

#### c) Casos de Uso
```
Tienes múltiples servicios en mismo APP Gateway:
- /images/* → Pool de servidores de imágenes
- /videos/* → Pool de servidores de videos
- Todo en 1 Application Gateway
```

#### d) Beneficios
- **Auto-scaling**: Escala según tráfico automáticamente
- **End-to-end encryption**: Tráfico encriptado completo
- **Zone redundancy**: Span múltiples availability zones
- **Multi-site hosting**: Hasta 100 websites en 1 instance
- Reduce costos y complejidad

### 5. ExpressRoute (Conexión Directa a Azure)

#### a) Problema
```
VPN funciona pero:
  - Va por internet público
  - Menos segura
  - Más lento
Necesito conexión: PRIVADA + RÁPIDA + SEGURA
→ ExpressRoute
```

#### b) Definición
- Conexión **private + dedicated** de datacenter on-premises → Azure
- NO va por internet público
- Más confiable, más rápido, más bajo latency

#### c) Características
- **Privada**: No comparte internet público
- **Alta disponibilidad**: Conecta confiable
- **Alto ancho de banda**: No limitado por internet
- **Bajo latency**: Conexión directa

#### d) Casos de Uso
- Datos deben ser on-premises + periodically migrados
- High-availability requirement
- On-premises ↔ Azure data needs to be frequent

### 6. Content Delivery Network (CDN)

#### a) Problema
```
Mi website está en Australia
Usuarios en París quieren acceder
Latencia alta: Tienen que esperar

¿Cómo hago que sea rápido globalmente?
→ Azure CDN copia contenido worldwide
```

#### b) Definición
- **Distributed network** de servidores
- Almacena cached content en **edge servers** (cerca de usuarios)
- Minimiza latencia

#### c) Arquitectura
```
Origin Server: Australia
    ↓
Edge Nodes (worldwide):
- Paris node
- Tokyo node
- São Paulo node
- etc.

Usuario request:
1. Pide al node más cercano
2. Si node tiene data (no expirada): Devuelve rápido
3. Si expirada: Pide fresh data a origin, cachea, devuelve
```

#### d) Caching y Invalidación
- Cada pieza de contenido tiene **expiry date**
- Típicamente: horas (depende de frecuencia de cambio)
- Si expira: Pide fresh copy del origin
- Proceso: Invisible al usuario, muy rápido

#### e) Beneficios CDN
- **Mejor performance**: Usuarios ven data desde node cercano
- **Mejor UX**: Websites cargan más rápido
- **Global scaling**: Maneja high loads instantáneament (ej: product launch)
- **Reduce backend**: Edge servers protegen tu origin server
- **Distribuye requests**: Menos traffic al origin

#### f) Terminología CDN
- **Cache**: Colección de copias temporales de files
- **Origin Server**: Original/master copy de tu app (Australia en ejemplo)
- **Expiry/TTL**: Cuándo cache expire

### 7. Comparación de Servicios de Networking

| Servicio | Función | Cuándo Usar |
|----------|---------|-----------|
| **VNet** | Red privada | SIEMPRE (base de todo) |
| **Load Balancer** | Distribuye por IP+port | Múltiples VMs, mismo app |
| **App Gateway** | Distribuye por HTTP properties | Múltiples apps/servicios |
| **VPN Gateway** | Conexión secure hybrid | On-premises + Azure |
| **ExpressRoute** | Conexión private dedicated | Mission-critical, frequent data |
| **CDN** | Distribuye contenido globalmente | Media, assets, performance |

## 📚 Conclusión y Síntesis Profunda - Capítulo 05: Networking

### ¿Qué Aprendiste Realmente?

**Networking es cómo hablan las cosas.** 

Sin networking, tienes VMs aisladas. Con networking, tienes VMs comunicándose, usuarios accediendo, datacenters replicándose.

El capítulo cubrió la pregunta fundamental: **"¿Cómo comunico recursos Azure entre ellos y con el mundo?"**

### Virtual Networks: El Fundamento

**VNet es universal.** TODA aplicación en Azure necesita estar en una VNet. Es como el espacio virtual donde viven todos tus recursos.

```
Antes (On-Premises):
Server A ← Cable físico → Switch → Cable físico → Server B

Ahora (Azure - VNet):
Server A ← Código → Switch Virtual (Azure) → Código → Server B
```

Es virtualizado, pero los conceptos son idénticos.

**Address Space + Subnets:**

```
VNet Address Space: 10.0.0.0/16
└─ Subnet 1 (10.0.1.0/24): Frontend (Usuarios externos acceden)
└─ Subnet 2 (10.0.2.0/24): Backend (Computación)
└─ Subnet 3 (10.0.3.0/24): Database (Datos sensibles)
```

¿Por qué segmentar? **Seguridad y aislamiento.**

```
Si attacker compromete Subnet 1 (Frontend):
└─ Firewall rules NO les permiten acceder Subnet 3 (Database)
└─ Data está segura
```

### Load Balancer vs Application Gateway: "¿Dónde Distribuyes?"

Aquí es donde muchos confunden. **Son diferentes herramientas para diferentes problemas.**

```
LOAD BALANCER ("Layer 4")
└─ Distribuye tráfico por IP + Puerto
└─ Rápido (poco análisis)
└─ Uso: Múltiples VMs con misma aplicación
   Ejemplo: Tienes 5 VMs web. LB rota entre ellos.

APPLICATION GATEWAY ("Layer 7")
└─ Distribuye tráfico por HTTP properties
└─ Más inteligente (entiende HTTP)
└─ Uso: Múltiples APLICACIONES diferentes en mismo gateway
   Ejemplo:
   - /images/* → Pool de servidores de imágenes
   - /videos/* → Pool de servidores de videos
   - /api/* → Pool de servidores de API
```

**Ejemplo práctico:**

```
Company tiene website con 3 secciones:
1. Carrito de compras (reqs pesadas, necesita muchos servers)
2. Blog (requests livianos, 1 server alcanza)
3. Admin Panel (requests raros, 1 server alcanza)

Con Load Balancer:
└─ 5 VMs idénticas (carrito + blog + admin todo)
└─ 3 VMs subutilizadas (wasting dinero en blog/admin)

Con Application Gateway:
└─ 3 VMs para carrito
└─ 1 VM para blog
└─ 1 VM para admin
└─ Gateway redirige /carrito/* → Pool 1, /blog/* → Pool 2, etc
└─ Dinero ahorrado
```

### VPN Gateway: Hybrid Cloud

Es la respuesta a: "Tengo datacenter on-premises. Tengo servers en Azure. ¿Cómo los comunico seguramente?"

```
Opción A: Direct internet
└─ Pros: Simple
└─ Cons: ¡Cualquiera puede interceptar datos!

Opción B: VPN Gateway
└─ Crea "encrypted tunnel" por internet público
└─ Data viaja encriptada
└─ Incluso si interceptan, es gibberish
└─ Como envío una carta en caja cerrada vs. abierta
```

**Arquitectura:**

```
On-Premises
└─ VPN Gateway (tu lado)
   └─ Encrypted tunnel
      └─ Internet público
         └─ VPN Gateway (Azure lado)
            └─ Azure VNet
```

### ExpressRoute: "Premium VPN"

VPN es bueno pero tiene límites:

```
VPN:
└─ Va por internet público
└─ "Performance es suficiente"
└─ Barato
└─ Conexión no garantizada

ExpressRoute:
└─ Conexión PRIVADA y DEDICADA
└─ Bajo latencia garantizado
└─ Rápido (hasta 100Gbps)
└─ No va por internet
└─ Caro (~$1000+/mes)
```

**Cuándo usar:**

```
VPN: "Empleados remotos acceden aplicació"
ExpressRoute: "Migración de datacenter completo. Necesito ancho de banda garantizado"
```

### Content Delivery Network (CDN): Ir Global

Imagina:

```
Tu website en Australia
Usuarios en Paris solicitan

Paris → Internet → Australia (300ms latencia)
       → Usuarios esperan
       → Usuarios se van
```

**Solución: CDN**

CDN duplica tu contenido en servidores distribuidos globalmente:

```
Tu website original: Australia
CDN Edge Servers:
├─ Paris (para usuarios Europa)
├─ Tokyo (para usuarios Asia)
├─ Nueva York (para usuarios USA)
├─ São Paulo (para usuarios Sudamérica)
```

Cuando usuario en París solicita:
1. CDN local en París: "¿Tengo esto en caché?" SÍ → Devuelve rápido (10ms)
2. Si NO (contenido expirado): Pide a origin (Australia), cachea, devuelve

**Resultado:**
```
Sin CDN: 300ms + latency = Lento
Con CDN: 10ms + latency = Rápido
```

**Casing Complexity:**

```
TTL (Time To Live) = "Cuánto tiempo caché es válido"
└─ Imagen de producto: 30 días (cambio raramente)
└─ Última noticia: 5 minutos (cambios frecuentes)
└─ Stock de acciones: 5 segundos (cambian constantemente)

Azure decide cuándo refrescar automáticamente.
```

### VNet Peering: Comunicar VNets

Si tienes múltiples VNets (diferentes proyectos, diferentes equipos), necesitan comunicarse.

```
VNet A (Frontend)  ←  VNet Peering  → VNet B (Backend)
```

**Sin peering:**
```
A → Internet → B (¡Inseguro! Data por internet público)
```

**Con peering:**
```
A ← Backbone privado Microsoft → B (Seguro, privado)
```
