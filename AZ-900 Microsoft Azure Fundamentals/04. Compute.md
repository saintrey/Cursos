# 04. Compute (Computación)

## Resumen del Capítulo
Compute es el corazón de Azure. Cubre todas las formas de ejecutar aplicaciones en la nube: desde máquinas virtuales tradicionales hasta funciones serverless. Incluye IaaS (VMs), PaaS (App Services) y Serverless (Functions).

## Temas Principales

### 1. Virtual Machines (VMs - Máquinas Virtuales)

#### a) Definición
- **Computadora virtual** que tienes acceso EXCLUSIVO
- No posees/administras hardware físico (Azure sí)
- **IaaS**: Tú administras TODO excepto el hardware físico

#### b) Ventajas
- Acceso exclusivo a la máquina
- Escalabilidad: Crea/elimina rápidamente
- Herramientas Azure para administra miles de VMs
- Soporte multi-plataforma: Windows o Linux
- Recomendaciones automáticas de Azure

#### c) Casos de Uso
- Necesitas control total de la máquina
- Instalar aplicación específica
- Migrar infraestructura on-premises
- Cuando otros servicios (App Services) no encajan

#### d) Cuándo NO usar VMs
- Si App Services/PaaS cubre tu caso (menos mantenimiento)
- El mantenimiento es responsabilidad tuya (pain point)

#### e) Precios
- **Calculado por hora**
- Depende de CPU y RAM (a más, más caro)
- **Nuevas opciones**: GPUs especializadas
- GPU para: AI, Deep Learning

#### f) Características Azure para VMs
- **Availability Sets**: Grupo de VMs para alta disponibilidad
- **Blueprints**: Templates de cómo crear VMs (consistency)
- **Azure Advisor**: Recomendaciones automáticas para HA, seguridad, performance, costos

### 2. Scale Sets (Conjuntos de Escalado)

#### a) Definición
- **Grupo de VMs idénticas**, load-balanced
- Crea clones a partir de una VM base
- Auto-scaling: Agrega/elimina automáticamente

#### b) Analogía del Vaso de Agua
```
1 VM = 1 vaso a 50% capacidad (CPU 50%)
Si sube a 90%:
  ↓
Agrega otro VM automáticamente = 2 vasos a ~45% cada uno
Si baja la demanda:
  ↓
Elimina vasos automáticamente (no pagas por recursos no usados)
```

#### c) Características
- **Idénticas**: Todas las VMs en scale set son iguales
- **Load Balanced**: Distribuye tráfico automáticamente
- **Auto-Scaling**: Basado en reglas o cpu/memoria
- **Scheduled Scaling**: Escala a horas específicas

#### d) Beneficios
- Fácil gestión de múltiples VMs
- Consistency (todas iguales)
- Load balancing integrado
- **Alta Disponibilidad**: Si 1 VM falla, otras sirven
- Auto-scaling reduce costos
- Escalable hasta 1000 VMs en un scale set
- **Sin costo extra**: Pagas solo por VMs + storage + load balancers

#### e) Caso de Uso Real
- E-commerce: Tráfico sube noches/weekends
- Scale Sets automáticamente agrega VMs
- Cuando tráfico baja, elimina VMs
- Result: Mejor performance + costos optimizados

### 3. App Services

#### a) Definición
- **Platform as a Service (PaaS)** completamente managed
- Azure administra: servidores, red, storage, infraestructura
- Tú administras: tu lógica de negocio

#### b) Tres Tipos de App Services

**1. Web Apps**
- Websites y aplicaciones online típicas
- Soporta: .NET, Java, Node.js, PHP, Python (Windows) o .NET Core, Node.js, PHP, Ruby (Linux)
- Integración con herramientas Azure/Microsoft
- Auto-scaling y load balancing
- Fácil deployment

**2. Web Apps for Containers**
- Despliega aplicaciones containerizadas
- Containers = paquete de app + todas sus dependencias
- Portable: Funciona en cualquier lugar con container runtime
- Analogía: Nintendo Cartridge (todo lo que necesitas adentro)

**3. API Apps**
- Backend data services sin frontend gráfico
- Application Programming Interface
- Conecta otras apps programáticamente
- Lenguajes: C#, Python, PHP, Java
- Acceso seguro integrado

#### c) Ventajas App Services
- Fully managed: No worry about hardware
- Multi-lenguaje support
- Built-in auto-scaling + load balancing
- Fácil de usar

### 4. Azure Container Instances (ACI)

#### a) Qué es un Container
- **Self-contained unit** de software
- Incluye: código, programas, todas las dependencias
- **Portable**: Funciona en cualquier máquina con container runtime
- Analogía: Envío de paquetes en containers estándar vs. custom packaging
- Resultado: Consistencia entre ambientes

#### b) Analogía Detallada del Envío
```
Sin Containers (Tradicional):
  - Cada item necesita custom packaging
  - Lento, error-prone
  (Como los desarrolladores diciendo "funciona en mi máquina")

Con Containers:
  - Todo en container estándar
  - Funciona en cualquier lugar
  - Consistencia garantizada
```

#### c) Beneficios Containers
- **Manejabilidad de Dependencias**: Todo adentro del container
- **Menos Overhead**: Comparado con VMs
- **Portabilidad**: Deploya en cualquier lugar
- **Efficiency**: Ciclo desarrollo/deployment más rápido
- **Scaling**: Más fácil escalar containers que VMs
- **Consistency**: Mismo comportamiento everywhere

#### d) Azure Container Instances (ACI)
- Servicio Azure para hostear/correr containers
- Sin administrar VMs
- On-demand: Crea container solo cuando lo necesites, después apaga
- Perfecto para: Procesamiento data bajo demanda
- Acceso: Portal, CLI, PowerShell

### 5. Azure Kubernetes Service (AKS)

#### a) Qué es Kubernetes
- **Orchestration system** open-source por Google (nombre griego: "govern")
- Automatiza: deployment, scaling, management de containers
- **No es Azure-specific**: Kubernetes es universal

#### b) Funcionalidades Kubernetes
- **Automatic Deployment**: Despliega instancias automáticamente
- **Automatic Scaling**: Monitorea carga, escala según sea necesario
- **Replicable Architecture**: Reutiliza setups probados

#### c) Azure Kubernetes Service (AKS)
- **Managed** Kubernetes por Azure
- Includes: All Kubernetes features + Azure services
- **Cluster**: Set de máquinas (nodes) que corren containers
- **Pod**: Grupo de 1+ containers con storage/network compartidos
- Integración: Identity Management, elastic provisioning, VS Code, global reach

#### d) Azure Container Registry (ACR)
- Almacena container images (como un registro)
- Cuando AKS/ACI necesita imagen, viene de ACR
- Seguridad Azure integrada
- Es como "app store" de containers

#### e) Caso de Uso Kubernetes
```
App procesa órdenes de clientes → Container
Más órdenes → Kubernetes automáticamente crea more Pods
Fewer órdenes → Kubernetes elimina Pods innecesarios
```

### 6. Azure Virtual Desktop (AVD)

#### a) Definición
- Windows 10 completamente virtualizado 100% en Azure
- Acceso desde: Windows, Mac, iOS, Linux, smartphones
- No corre en tu máquina, corre en Azure

#### b) Casos de Uso
- Empleados remotos: Rápido acceso a Windows en cloud
- Usuarios on-premise o on-premises+cloud
- Multi-user: Múltiples usuarios en misma instancia

#### c) Ventajas
- **Licensing**: Reutiliza Windows 10 existentes (reduce costos)
- **Multi-Session**: Múltiples usuarios = instancia
- **Anywhere Access**: De qualquier dispositivo, ubicación
- **Data Security**: Data en Azure, no en dispositivo
- **Integration**: Azure security features (MFA, RBAC, Policies)

### 7. Azure Functions (Serverless)

#### a) Definición
- **Smallest + largest** compute service Azure
- **Función única** de computación
- Se ejecuta cuando su URL es llamada (invoked)
- Después de completarse: Se muere (sin persistencia)
- **TRUE Serverless**: No administras servers en absoluto

#### b) Arquitectura
```
User hits function URL
    ↓
Function triggered
    ↓
Executes once
    ↓
Talks to other services / returns response
    ↓
Function ends (dies)
```

#### c) Caso de Uso: FunnyFerret Business
```
Usuario sube imagen de ferret graciosa
    ↓
Azure Function:
  - Comprime imagen
  - Convierte a formato web
  - Almacena en base de datos
    ↓
Function completa y muere
```

#### d) Ventajas Functions
- **Costo**: Pagas solo cuando se ejecuta (no running 24/7)
- **Resilience**: Si 1 function falla, otros no se ven afectados
- **Isolation**: Cada función aislada (risk spread)

#### e) ACG Infraestructura
- **Todo serverless**: Cero VMs en ACG
- Permite crecimiento flexible
- Sirve cursos a audiencia masiva
- Trend continuo de serverless

## Comparación de Compute Services

| Servicio | Tipo | Control | Mantenimiento | Scaling | Costo |
|----------|------|---------|-------------|---------|-------|
| VM | IaaS | Alto | Alto | Manual | Por hora |
| Scale Sets | IaaS | Alto | Alto | Auto | Por hora |
| App Services | PaaS | Medio | Bajo | Auto | Según plan |
| Containers | Micro | Medio | Medio | Auto | Según uso |
| Kubernetes | Orquestr | Alto | Medio | Auto | Según recursos |
| Functions | Serverless | Bajo | Ninguno | Auto | Por ejecución |

## 📚 Conclusión y Síntesis Profunda - Capítulo 04: Compute

### ¿Qué Aprendiste Realmente?

**Compute es donde VIVE tu código.** Aprendiste que no existe "un único contenedor de computación"—existe un ESPECTRO, desde máximo control (VMs) hasta cero control (Functions).

El hilo conductor de todo este capítulo: **"¿Cuánto control necesito? ¿Cuántos detalles administrar?"**

```
Controls Spectrum:

VM (IaaS)
┌─────────────────────────────────────────────────────────────┐
│ TÚ administras: Patches, SO, runtime, dependencias, scaling │
│ Azure administra: Hardware                                    │
│ Ventaja: Control TOTAL                                        │
│ Desventaja: Mantenimiento TOTAL                               │
└─────────────────────────────────────────────────────────────┘
                              ↓
         App Services (PaaS - Managed)
┌─────────────────────────────────────────────────────────────┐
│ TÚ administras: Tu código solamente                            │
│ Azure administra: SO, runtime, patches, escalado               │
│ Ventaja: Menos complejidad                                     │
│ Desventaja: Menos flexibility                                  │
└─────────────────────────────────────────────────────────────┘
                              ↓
         Functions (Serverless)
┌─────────────────────────────────────────────────────────────┐
│ TÚ administras: Función única                                  │
│ Azure administra: Absolutamente TODO                           │
│ Ventaja: Zero operational overhead                             │
│ Desventaja: Solo funciones pequeñas                            │
└─────────────────────────────────────────────────────────────┘
```

**No hay "mejor."** Hay "correcto para el caso."

### Virtual Machines: "Cuando Necesitas Control Total"

VMs es donde comenzaron los computadores modernos. "Alquile un servidor."

**Cuándo usar:**
- Necesitas software específico que requiere mucha customización
- Necesitas compiladores, IDEs, bases de datos legacy
- Migrando on-premises (aplicación vieja que no refactorizarás)

**Cuándo NUNCA usar:**
- "Solo necesito ejecutar Node.js"—App Services es mejor
- "Solo necesito procesar data ocasionalmente"—Functions es mejor

**La verdad incómoda:** VMs es 80% tedium. El 80% de tu tiempo es:
- Applying patches de seguridad
- Monitorearing disk space
- Debugging SO issues
- Actualizaciones de runtime

Ese 80% tedium debería ser hecho por Azure (i.e., PaaS). Así que si puedes elegir, elige PaaS.

**Pero a veces no PUEDES.** Entonces VMs.

### Scale Sets: "VMs Que Se Clonan Solos"

Scale Sets es la respuesta a "tengo una VM funcionando perfectamente. ¿Cómo manejo 10x más tráfico sin reconfiguring manualmente 10 nuevas VMs?"

**Idea:**
- Tienes 1 VM perfecta configurada
- Scale Set clona eso a 10 VMs idénticas
- Load Balancer distribuye tráfico entre los 10
- Si sube demanda: "Agreg 5 más"
- Si baja: "Eliminar 5 que no uso"

**Resultado:** Tu app escala automáticamente sin interacción manual.

```
Noche de Black Friday (tráfico 10x):
└─ Auto-Scaling detecta CPU > 80%
   └─ Automáticamente crea 5 VMs nuevas
   └─ Load Balancer redirige tráfico
   └─ Usuarios ven performance normal
   └─ Tú duermes

Día siguiente (tráfico normal):
└─ Auto-Scaling detecta CPU < 40%
   └─ Automáticamente elimina 5 VMs innecesarias
   └─ Ahorras dinero
```

**Lección:** Scale Sets no es "feature extra"—es essencial si tienes aplicación con tráfico variable. Sin ello, o provisiona para peak (costoso) o usuarios se frustran (mal negocio).

### App Services: "El Punto Dulce para Muchas Apps"

App Services es Azure diciendo "oye, sabemos que 80% de aplicaciones necesitan lo mismo. Tomamos eso 80%, lo empaquetamos bonito, y lo damos a bajo precio."

```
App Services Plan = "Nivel de capacidad"

Free/Shared: "Experimental"
  └─ No production
  └─ 1 GB RAM, CPU compartido
  └─ $0/mes  
  
Basic: "Pequeña producción"
  └─ 1 GB RAM, 1 CPU
  └─ $10-20/mes

Standard: "Producción típica"
  └─ 2-8 GB RAM, múltiple cores
  └─ $50-150/mes
  
Premium: "Alto performance"
  └─ 16+ GB RAM, dedicated instance
  └─ $200+/mes
```

**Por qué App Services > VMs para web apps:**

```
VM approach:
1. Provee VM (20 min)
2. Instala runtime (10 min)
3. Deploy app (5 min)
4. Setup SSL (10 min)
5. Configure autoscaling (15 min)
Total: 1 hora

App Services approach:
1. Deploy app
Total: 5 minutos
```

Aparte que App Services:
- Patches automáticos
- Escalado automático
- SSL integrado
- Integración CI/CD (GitHub Actions, etc)
- Monitoreo integrado

**Lección:** Si tu app es "código + runtime" (Node, Python, .NET, Java), probablemente App Services. Solo elige VMs si realmente necesitas eso 20% de control extra.

### Containers: "Solución del "Funciona en Mi Máquina""

Este es uno de los problemas más irritantes en software:

```
Dev: "Funciona en mi máquina"
QA: "No funciona aquí"
Prod: "Se crashea"

Razones:
- Dev tiene Python 3.8, Prod tiene Python 3.6
- Dev instaló librerías globales, Prod no
- Dev usa Windows, Prod usa Linux
- Networking diferente
```

**Solución: Containers**

Containers es "paquete tu app + TODAS sus dependencias + el runtime exacto que necesita."

```
Container = Caja cerrada
└─ App code
└─ Python 3.8 exactamente
└─ Todas las librerías exactamente
└─ Configuración exacta de networking
```

Llevas esa caja de dev, la ejecutas en prod → **Exactamente lo mismo.**

**Así que:**
- Dev: "Puse app en Docker container"
- QA: "Ejecuto ese container" = Exactamente igual que dev
- Prod: "Ejecuto ese container" = Exactamente igual que dev y QA

**Problema resuelto.** 

**Azure Container Instances (ACI):**
- "Ejecuta mi container en Azure"
- No piensas en VMs
- Azure maneja la infraestructura
- Paga por segundo

**Caso de uso:** Task que corre cada noche. Normalmente:
- "Debo tener una VM ejecutándose 24/7" = Costoso
- Con container: "Ejecuto container a las 10pm, termina a 10:15pm, pago 15 minutos"

### Kubernetes y AKS: "Orquestación de Containers"

Si tienes 1 container, es simple. ¿Pero si tienes 50 containers que necesitan comunicarse, replicarse, balancearse?

**Kubernetes es el orquestador:**

```
Tú deseas: "Wants 10 instances de mi API container"
Kubernetes: "Desplegué 10. Monitor salud. Si 1 falla, creo 1 nuevo.
            Si suben usuarios, escalo a 15. Si bajan, escalo a 5."
```

Kubernetes es open-source (no es de Azure). Mas es COMPL COMPLEJO (100+ conceptos).

**AKS (Azure Kubernetes Service) = Managed Kubernetes**

Azure dice "Kubernetes es poderoso pero complejo. Tomamos complejidad, dejas simplificad:"

```
Normal Kubernetes:
- Administras cluster
- Administras nodes
- Administras networking
- Parchea todo
- 80% administración

Azure AKS:
- Te preocupas solo containers
- Azure: Cluster, nodes, patching, networking
- 20% administración
```

**Cuando usar AKS:**
- Aplicación compleja con muchos servicios interdependientes
- Netflix, Uber, Spotify escala

**NO uses AKS si:**
- Tienes 1 app simple
- Tienes pequeño equipo
- "Necesito Kubernetes porque escuchécomo es cool"

AKS es como "me compré un Ferrari." Si manejabas un Civic, necesitas aprender a conducir Ferrari. Si no necesitas Ferrari, ¿por qué los problemas?

### Azure Functions: "Serverless Verdadero"

**Serverless NO significa "sin servidores."** Significa "no administras servidores."

Functions es la unidad **MÁS PEQUEÑA** de computación en Azure:

```
VM = Aplicación entera ejecutándose continuamente
App Service = Aplicación ejecutándose continuamente
Function = Una función que corre cuando se invoca, entonces muere
```

```
Función triggerada por HTTP:

User makes request
    ↓
Request hits Azure
    ↓
Azure: "Oh, necesita ejecutar function X"
    ↓
Azure: ENCIENDE serverless instance
    ↓
Function ejecuta (200ms)
    ↓
Azure: APAGA serverless instance
    ↓
User recibe respuesta

Costo: Pagas por 200ms + execuciones (muy muy barato)
    vs. VM running 24/7 = Caro
```

**Casos de uso:**
- Webhooks (GitHub pushea, trigger función)
- Data processing (S3 sube archivo, función procesa)
- Scheduled tasks (Cron: "cada 1am ejecuta limpieza")
- Integración (Slack → Function → Database)

**Ventaina: Pagas por uso exacto. Si función no se ejecuta, pagas $0.**

**Desventaja: No puedes guardar state entre ejecuciones. Cada ejecución es "fresca."**

```
// NO hagas esto:
static counter = 0  // ← No funciona. Se resetea cada ejecución

function() {
  counter++  // Siempre será 0+1 = 1
}
```

Para state, necesitas database/storage separado.

### Eligiendo La Herramienta Correcta

Final "decision tree":

```
¿Necesitas máquina completa?
  └─ SÍ → VM
  └─ NO → App Service

¿Tu app es funciones pequeñas?
  └─ SÍ → Functions
  └─ NO → App Service

¿Escalas de 1 a 1000 usuarios unpredictably?
  └─ SÍ → App Service + auto-scaling
  └─ SÍ (containers) → AKS

¿Necesitas rápidamente procesar data sin server running?
  └─ SÍ → Functions O ACI
```

### Resumen: Compute Es "dónde Escribes"

Compute es el lugar donde tú código **vive y ejecuta.** Sin compute, no hay aplicación.

Todo lo que aprendiste—VMs, App Services, Containers, Functions—es respuestas a "¿cómo ejecuto mi código?" con diferentes trade-offs.

La pregunta implícita siempre es: "¿Cuánto tedium administrativo puedo tolerar?"

- VMs: "100% tedium, 100% control"
- App Services: "10% tedium, 95% control" ← Punto dulce para mayoría apps
- Containers + Kubernetes: "20% tedium, 90% control" ← Complejidad con poder
- Functions: "0% tedium, 0% control" ← Perfecto para tareas pequeñas

Conocer ese spectrum significa que cuando el boss dice "necesitamos aplicación que escala automáticamente," sabes exactamente QUÉ elegir en lugar de adivinar.

**Ese es el expertise. 🚀**
