# 02. Cloud Concepts (Conceptos de Nube)

## Resumen del Capítulo
Este capítulo explora los fundamentos del concepto de computación en la nube, incluyendo los beneficios, el lenguaje técnico, los modelos económicos y los diferentes servicios disponibles en Azure.

## Temas Principales

### 1. Definición de Cloud Computing
- **Cloud Computing**: Disponibilidad bajo demanda de recursos de sistemas de computación divididos en tres categorías:
  - Computación
  - Red
  - Almacenamiento

- **Características principales**:
  - Bajo costo
  - Fácil adopción
  - Recursos prácticamente ilimitados
  - No hay necesidad de infraestructura física propia

### 2. Lenguaje de Cloud Computing

#### a) High Availability (Alta Disponibilidad)
- Capacidad de múltiples máquinas virtuales de reemplazarse automáticamente
- Si un servidor falla, otro disponible toma su lugar
- Asegura que servicios estén siempre disponibles
- No garantiza 0 downtime (mejor que eso es confiabilidad)

#### b) Reliability/Fault Tolerance (Confiabilidad/Tolerancia a Fallos)
- Sistema resiliente que se recupera automáticamente
- No significa 0 downtime necesariamente
- Se logra desplegando recursos en múltiples ubicaciones
- Ningún servidor actúa como punto único de fallo

#### c) Scalability (Escalabilidad)
- Proceso de agregar más recursos según sea necesario
- **Scaling Out** (Horizontal): Agregar más máquinas
- **Scaling Up** (Vertical): Hacer máquinas existentes más poderosas
- Auto-scaling: Ocurre automáticamente según usos o reglas definidas
- Ejemplo: Agregar más mugs (máquinas) o hacer mugs más grandes

#### d) Predictability (Previsibilidad)
- **Predictable Performance**: Aplicación funciona consistentemente sin importar cargas
- **Predictable Costs**: Sin sorpresas en facturas, puedes forecasr uso y costos

#### e) Security (Seguridad)
- Control total de la seguridad de recursos cloud
- Puede ser control total (IaaS) o delegado parcialmente (PaaS)
- Incluye patches, mantenimiento, control de tráfico de red

#### f) Governance (Gobernanza)
- Capacidad de establecer estándares corporativos
- Restricciones en recursos desplegados si es necesario
- Auditoría de recursos
- Cumplimiento normativo

#### g) Manageability (Mantenibilidad)
- **Management OF the Cloud**: Escalar automáticamente, visibilidad de salud, deployment con templates
- **Management IN the Cloud**: Cómo interactúas (Portal, CLI, APIs)

### 3. Economía del Cloud Computing

#### a) Capital Expenditure (CapEx)
- Dinero gastado en adquirir o mantener activos fijos
- Compra de servidores, edificios, equipos
- Costo único por adelantado

#### b) Operational Expenditure (OpEx)
- Costo continuo para operar día a día
- Cartuchos de impresoras, luz, servicios anuales
- Con cloud: pago mensual bajo como modelo pay-as-you-go

#### c) Consumption-Based Pricing (Precios Basados en Consumo)
- Pagas solo por lo que usas
- Por ejecución de función, por segundo, por combinación
- Perfecto para startups con poco tráfico inicial
- Ejemplos: Azure Functions, almacenamiento variable

### 4. Cloud Service Models (Modelos de Servicios Cloud)

#### a) Infrastructure as a Service (IaaS)
- Servidores virtualizados provistos por Azure
- **Tú administras**:
  - Software
  - Sistemas operativos
  - Middleware
  - Aplicaciones
- **Azure administra**: Hardware físico, virtualización
- Ejemplos: Virtual Machines, Storage, Networking, Firewall
- **Ventaja**: Control completo
- **Desventaja**: Mantenimiento completo

#### b) Platform as a Service (PaaS)
- Superset de IaaS + middleware, herramientas desarrollo, BD, BI
- Soporte para ciclo completo: build, test, deploy, manage, update
- **Tú administras**: Aplicaciones y servicios que desarrollas
- **Azure administra**: Casi todo más (seguridad, herramientas dev, BD)
- Ejemplos: Azure App Services, Azure CDN, Cosmos DB
- **Ventaja**: Menos complejidad, enfocarse en negocio
- **Desventaja**: Menos control

#### c) Software as a Service (SaaS)
- Servicio alojado completamente por proveedor
- Acceso mediante internet
- No posees el software, pagas por acceso
- Siempre última versión
- **Tú NO administras nada** relacionado a infraestructura
- Ejemplos: Office 365, Gmail, QuickBooks, Azure SQL
- **Ventaja**: Cero mantenimiento, siempre actualizado

#### d) Serverless
- NO significa que no hay servidores
- Significa que tú NO administras servidores
- Usas servidores de terceros
- Ejemplo: Azure Functions
- Función única de código se hospeda, deploya, ejecuta y gestiona sola

### 5. Modelo de Responsabilidad Compartida
| Aspecto | On-Premises | IaaS | PaaS | SaaS |
|--------|-------------|------|------|------|
| **Tú Responsable** | TODO | Software, SO, Middleware, Apps | Apps y Servicios | Datos y Dispositivos |
| **Microsoft Responsable** | Nada | Hardware, Red, DC | OS, Red, Apps PaaS, Identidad | Controles Red, Apps, Identidad |
| **Compartido** | - | - | Identidad | Identidad |

### 6. Azure Marketplace
- App Store para servicios cloud
- Soluciones de Microsoft y partners
- Tipos: SaaS, VMs, solution templates, managed apps, servicios de consultoría
- Categorías: SO, bases de datos, seguridad, identidad, networking, blockchain, dev tools
- **Acceso**: Web, CLI, Portal integrado
- **Beneficios**: Menos mantenimiento, certificado por Microsoft, prototipo rápido, vender tus productos

### 7. Cloud Architecture Models (Modelos de Arquitectura)

#### a) Private Cloud (Nube Privada)
- Servicios cloud en hardware propio
- **Ventajas**: Control total, seguridad y privacidad altos, beneficios de cloud
- **Desventajas**: TI responsable de todo, costos de staff/mantenimiento altos
- No aprovecha economías de escala

#### b) Public Cloud (Nube Pública)
- Azure, AWS, GCP, etc.
- Accesible para cualquiera (internet)
- **Ventajas**: No compras hardware, costos bajos, cloud provider responsable
- **Desventajas**: Poco control, actualizaciones impuestas, no sabes dónde está tu data
- Económico y escalable

#### c) Hybrid Cloud (Nube Híbrida)
- Mezcla de private y public cloud
- Migraciones gradualmente sin disrupciones
- Datos sensibles on-premises, tareas básicas en public
- **Ventajas**: Flexibilidad, poder, mejor CapEx que private
- **Desventajas**: Complrajidad alta, múltiples ubicaciones, muchas partes móviles

## Puntos Clave para el Examen

✅ **Recuerda**:
- **CapEx**: Pago adelantado, ahora raro en cloud
- **OpEx**: Pago continuo, modelo cloud moderno
- **IaaS**: Control total (VMs), tú administras SO y apps
- **PaaS**: Control medio (App Services), proveedor administra más
- **SaaS**: Control mínimo (Office 365), tú solo usas
- **Serverless**: No administras servers, paga por uso
- Escalabilidad ≠ Alta Disponibilidad (diferentes conceptos)
- Confiabilidad = Recuperación automática = 0 downtime

## 📚 Conclusión y Síntesis Profunda - Capítulo 02: Cloud Concepts

### ¿Qué Aprendiste Realmente?

Este capítulo es **el más importante del curso AZ-900**—aunque suene como "conceptos básicos," estos conceptos son lo que **diferencia a alguien que realmente entiende cloud de alguien que solo conoce Azure**.

Aprendiste que Cloud Computing no es "alquilar servidores en internet." Es un **cambio fundamental en cómo piensan los desarrolladores y empresas sobre infraestructura**. En lugar de predecir exactamente cuántos servidores necesitas (CapEx), ahora pagas por lo que usas cuando lo usas (OpEx).

### Los 3 Cambios Entre CapEx y OpEx (El Corazón del Cloud)

Imagina dos empresas:

**Empresa Vieja (CapEx)**
- Enero: Compra 10 servidores por $50,000
- Problema: ¿Y si en junio necesita 20? Tardará 3 meses en comprar, instalar, configurar
- Problema 2: ¿Y si en noviembre solo necesita 5? Gastó $50,000 en equipos que ahora acumulan polvo

**Empresa Nueva (OpEx - Cloud)**
- Enero: Levanta 10 VMs en Azure, paga $500/mes
- Junio: Levanta 10 VMs más, ahora paga $1000/mes
- Noviembre: Elimina 15 VMs, ahora paga $250/mes

Ves la diferencia? La empresa vieja está atrapada con decisiones de presupuesto. La empresa nueva es **ágil como el agua**—se adapta instantáneamente a la demanda.

**Esto es el corazón del cloud.** No es sobre la tecnología específica. Es sobre flexibilidad económica.

### IaaS vs PaaS vs SaaS - La Jerarquía de Responsabilidad

Después de cloudlear el modelo económico, viene lo más confuso: ¿cuándo usas cada uno?

Piénsalo así:

```
FULL CONTROL (IaaS)
You: Sistema operativo, aplicación, actualizaciones, patches, todo
Azure: Hardware físico
Caso: "Necesito instalar Oracle Database específico que requiere configuración muy particular"

BALANCE (PaaS)
You: Tu aplicación
Azure: Sistema operativo, runtime, base de datos, actualizaciones, patches
Caso: "Quiero desplegar mi Node.js app sin pensar en el servidor"

ZERO CONTROL (SaaS)
You: Nada
Azure: Todo
Caso: "Uso Office 365. Alguien más se preocupa por backups, security, patches"
```

La trampa es que **mientras MÁS CONTROL tienes, MENOS tiempo pasas escribiendo código de negocio.**

- IaaS: 80% administración, 20% código de negocio
- PaaS: 20% administración, 80% código de negocio
- SaaS: 0% administración, 100% código de negocio

Entonces, ¿por qué no usar SaaS siempre? **Porque no siempre encaja.** Si necesitas una base de datos NoSQL global con latencia sub-milisegundo, Office 365 no lo hace—necesitas Cosmos DB (PaaS) o el control total (IaaS).

### El Modelo de Responsabilidad Compartida - Tu Mapa de Seguridad

Ahí está la tabla que viste. **Memorízala de verdad.** No memorices "IaaS" y "PaaS"—entiende qué **TÚ** responsable de asegurar.

**Regla de oro**: Mientras MÁS bajo vayas en el stack (IaaS), más responsabilidad tienes por seguridad. Mientras MÁS alto (SaaS), más responsabilidad tiene Microsoft.

Caso práctico:
- Usas IaaS (VM): Un hacker ataca el SO. **¿Quién fallo en patchear? TÚ.**
- Usas SaaS (Office 365): Un hacker ataca Office. **¿Quién fallo en parchear? Microsoft.**

Eso es ENORME. Te ahorras un equipo de seguridad completo.

### Cloud Architecture Models - Elegir Tu Batalla

Escuchaste sobre Private, Public, Hybrid cloud.

**Private Cloud**: "Azure pero en mi datacenter"
- Caso: Banco suizo que NO PUEDE poner data en internet público
- Ventaja: Control total, compliance fácil
- Desventaja: Pagas por infraestructura + equipo + mantenimiento. Es como tener el peor de ambos mundos

**Public Cloud**: "Azure, AWS, GCP en internet"
- Caso: Startup con datos no-sensibles, o empresa con datos sensibles pero well-protected
- Ventaja: Barato, escalable, sin administración
- Desventaja: Menos control (pero con bien arquitectura, ¿lo necesitas?)

**Hybrid**: "Mezcla de ambos"
- Caso: Migración lenta on-premises → cloud
- Ventaja: Transición gradual, información sensible on-prem, información pública en cloud
- Desventaja: Complejidad ENORME. Dos sistemas. Dos equipos. Dos cosas que pueden fallar.

### Escalabilidad, Confiabilidad y Disponibilidad - Las Palabras Que Parecen Lo Mismo

Muchos confunden estas:

```
ESCALABILIDAD (Scaling)
¿Puedo manejar más carga?
Ejemplo: De 100 usuarios a 1000 usuarios
Herramienta: Scale Sets, auto-scaling

CONFIABILIDAD (Reliability)
¿Si algo falla, se recupera automáticamente?
Ejemplo: Si un servidor se apaga, otro automáticamente toma su lugar
Herramienta: Múltiples servidores en múltiples ubicaciones

DISPONIBILIDAD (Availability)
¿Cuál es el porcentaje de tiempo que está arriba?
Ejemplo: 99.9% availability = 43 minutos de downtime al año
Herramienta: Load balancers, failover

Analogía:
- Escalabilidad: "¿Puedo hacer que el restaurante maneje más clientes?"
- Confiabilidad: "¿Si el chef principal falta, otro chef prepara la comida?"
- Disponibilidad: "¿Estará abierto el restaurante mañana a las 12?"
```

Todos son DIFERENTES. Un sistema puede ser escalable pero NO confiable (una sola máquina grande se escala pero si falla = todo cae). Un sistema puede ser confiable pero NO disponible (si falla, tarda 2 horas en recuperarse, así que no estaba disponible). 

### Gobernanza y Seguridad - ¿Quién Decide Qué?

Este es un concepto que NO suena importante pero LO ES en el mundo real.

Imagina:
- Empresa con 5 departamentos
- Cada uno quiere usar Azure
- Cada uno elige configuraciones diferentes
- IT tiene pesadillas de seguridad/compliance

**Gobernanza** = "Aquí están las reglas. Todos deben seguirlas."
- "Todos los databases DEBEN estar encriptados"
- "Todos los VMs DEBEN tener Antimalware"
- "Solo pueden usar ciertas regiones (por compliance)"

Luego, **Seguridad** = "¿Estas reglas se están siguiendo?"

Juntos, significa que tu empresa no colapsa el día que auditores vienen a verificar compliance.
