# 06. Storage (Almacenamiento)

## Resumen del Capítulo
Storage es fundamental en Azure. Este capítulo cubre los diferentes tipos de almacenamiento disponibles, redundancia de datos, y métodos para transferir datos hacia/desde Azure.

## Temas Principales

### 1. Storage Account (Cuenta de Almacenamiento)

#### a) Definición
- **Namespace único de Azure** para todos tus datos
- Cada objeto almacenado tiene dirección web única
- Nombre de storage account es **globalmente único** en todo Azure
- Nadie más puede tener el mismo nombre

#### b) Naming Convention
```
Storage Account Name: "acloudguru"
   ↓
Web Address: acloudguru.blob.core.windows.net
             (para Blob Storage)

Importancia: Debe ser único globalmente
```

### 2. Blob Storage (Almacenamiento de Blobs)

#### a) Definición
- **BLOB**: Binary Large Object
- Cualquier cosa hecha de bits y bytes
- Versátil: Puede almacenar cualquier tipo de archivo

#### b) Estructura Jerárquica
```
Storage Account
└─ Blob Container (como carpeta)
   ├─ Blob 1 (archivo type1)
   ├─ Blob 2 (archivo type2)
   └─ Blob 3 (archivo type3)
```

#### c) Características
- **Flexible**: Diferentes tamaños, formatos
- **Acceso directo**: Cada blob tiene URL única
- **Labeling**: Sensible naming para encontrar rápido
- **Mixed content**: Imágenes, videos, documentos, todo

#### d) Tres Tipos de Blobs

**1. Block Blobs**
- Texto + datos binarios
- Hasta ~4.7 terabytes
- Compuesto de blocks individuales (manejables)
- Mejor para: Streaming, backups, cualquier cosa general

**2. Append Blobs**
- Tipo Block pero optimizado para append operations
- Ideal para: Logging (escribir líneas continuamente)
- Ejemplo: VM logs que crecen

**3. Page Blobs**
- Random access files
- Hasta 8 terabytes
- Acceso a cualquier parte de file anytime
- Uso principal: **Virtual Hard Drives** para VMs

#### e) Pricing Tiers (Capas de Precios)

**1. Hot Tier**
- Acceso frecuente
- Bajo latency
- **Costo**: Alto (acceso caro)
- Mejor para: Datos que accesas constantemente

**2. Cool Tier**
- Acceso menos frecuente
- Alto latency comparado a Hot
- Mínimo 30 días en Cool Tier
- **Costo**: Bajo storage, acceso más caro
- Mejor para: Backup, archivos ocasionales

**3. Archive Tier**
- Acceso muy raro
- Alto latency (esperas a recuperar)
- **Costo**: Más bajo storage disponible
- Mejor para: Datos históricos, compliance archival

### 3. Disk Storage (Almacenamiento de Discos)

#### a) Definición
- **Managed Disk**: Disco que atas a tu VM
- "Managed" = Azure administra hardware + garantiza uptime + backup

#### b) No Necesitas Preocuparte Por
- Tamaño físico del disco
- Performance
- Todo manejado por Azure

#### c) Cuatro Tipos de Discos

**1. HDD (Hard Disk Drive)**
- Spinning disk (viejo)
- **Costo**: Bajo
- **Uso**: Infrequent access, backup, testing
- **Latency**: Alto

**2. Standard SSD**
- Solid State Drive
- **Costo**: Medio
- **Uso**: Producción típica
- **Latency**: Bajo comparado a HDD
- **Confiabilidad**: Mejorada
- **Scalability**: Mejor que HDD

**3. Premium SSD**
- Super rápido
- **Costo**: Alto
- **Uso**: Cargas críticas (databases)
- **Latency**: Muy bajo
- **Performance**: Excelente

**4. Ultra Disk**
- Top tier, unicorns included 🦄
- **Costo**: Muy alto
- **Latency**: Sub-millisecond (increíble)
- **Size**: Hasta 64 terabytes
- **Uso**: Transaction-heavy, analytics complex, gaming, rendering

### 4. File Storage (Almacenamiento de Archivos)

#### a) Definición
- **Managed file storage** en cloud
- Comparte characteristics con file shares on-premises
- Extiende file sharing a la nube

#### b) Problemas Que Resuelve
- Espacio de disco on-premises limitado
- Backups deben ser configurados manually
- Seguridad de datos compleja
- Extender file sharing a teams remotas es complicado

#### c) Solución Azure Files
- **Escalabilidad**: Espacio unlimited
- **Managed**: Azure administra backups
- **Seguridad**: Azure security features
- **Remote Access**: Desde cualquier lugar

#### d) Casos de Uso
- Espacio on-premises casi lleno → Complementa con Azure Files
- Reemplaza on-premises file storage completamente
- **Lift and Shift**: Move on-premises → Azure gradualmente
- Data siempre available + resiliente

#### e) File Share En Azure
- SMB (Windows) y NFS (Linux) soportados
- Acceso como si fuera red drive local
- Performance + confiabilidad cloud

### 5. Archive Storage (Tier de Archivo)

#### a) Definición
- **Lowest cost** storage tier (por mucho)
- Terabytes de data por pocos dólares mensuales
- No es rápido ni performant
- No necesita ser rápido

#### b) Casos de Uso
- Compliance requirements (guardar por ley)
- Disaster recovery archives
- Historical data nunca accesido

#### c) Ventajas
- **Costo**: Extremadamente bajo
- **Durability**: Seguro y encrypted
- **Frechenimiento**: Libera valuable on-premises storage
- **Access**: Se mantiene on-premises, pero remote

### 6. Storage Redundancy (Redundancia de Almacenamiento)

#### a) Problema Que Resuelve
- Hardware falla (discos corruptidos)
- Data centers tienen outages
- Necesitas protección

#### b) Solución: Replicación Automática
- Azure SIEMPRE crea múltiples copias
- Automático e invisible
- **Mínimo**: 3 copias en región primaria
- Con paired regions: 6 copias totales (3 primaria + 3 secundaria)

#### c) Cuatro Opciones de Redundancia

**1. LRS (Locally Redundant Storage)**
- 3 copias en SINGLE datacenter/zone
- **Costo**: Bajo
- **Protección**: Disk failure en mismo rack
- **Limitación**: No protege zone outage
- Usar cuando: Costo primaria

**2. ZRS (Zone Redundant Storage)**
- 3 copias SPAN en 3 DIFFERENT zones (misma región)
- 1 copia por zona
- **Costo**: Medio
- **Protección**: Zone failure
- **Limitación**: No protege region-wide outage
- Usar cuando: Necesitas zone HA pero misma región

**3. GRS (Geo-Redundant Storage)**
- 3 copias en región primaria (LRS) + 3 en matched región (LRS)
- 6 copias totales
- **Costo**: Alto
- **Protección**: Region-wide failure
- **Limitación**: Solo 1 zone en primary (3 copias mismo lugar)

**4. GZRS (Geo-Zone-Redundant Storage)**
- 3 copias SPAN 3 zones en región primaria + 3 en matched región
- **Costo**: Muy alto
- **Protección**: Zone failures + región failures
- **Máxima redundancia**: Para datos críticos

#### d) Matriz de Redundancia

| Opción | Copias Primaria | Copias Secundaria | Protege | Costo |
|--------|-----------------|-------------------|---------|-------|
| LRS | 3 (1 zona) | - | Disk failure | Bajo |
| ZRS | 3 (3 zonas) | - | Zone failure | Medio |
| GRS | 3 (1 zona) | 3 (1 zona) | Región failure | Alto |
| GZRS | 3 (3 zonas) | 3 (1 zona) | Zone + Región | Muy Alto |

#### e) Read Access de Secundaria
- Enabled por default en G*RS
- Si primary región falla, read desde secondary rápidamente
- Failover automático y rápido

### 7. Moviendo Data (Data Migration)

#### a) Transferencias Pequeñas/Ocasionales

**1. AzCopy**
- Command-line utility
- Descargable, integra con terminal
- Soporta: Blob + Azure Files
- Ideal para: Scripting, automatización (Bash/PowerShell)
- Ejemplo: `azcopy cp video.mp4 mystorageaccount/mycontainer`

**2. Storage Explorer**
- GUI (Graphical User Interface)
- Descargable, user-friendly
- Drag-and-drop Como Windows Explorer
- Soporta: TODOS los storage types (no solo Blob/Files)
- Mejor para: Usuarios no-técnicos

**3. Azure File Sync**
- Sincroniza Azure Files ↔ On-premises file servers automáticamente
- Ideal para: Disaster recovery, multi-location sync, hybrid transition
- Casos: Remote workers, gradual cloud migration

#### b) Transferencias Grandes/Bulk

**1. Azure Data Box**
- Física encrypted hard drive
- Envían a tu ubicación
- Copias data a Data Box
- Envías de vuelta a Azure
- Azure copia al storage account
- Trabajo inverso también posible

Casos:
- Bulk migration > 80 terabytes
- Disaster recovery (remote backup)
- Security requirements (data too sensitive for internet)

**2. Azure Migrate**
- Suite de herramientas para migración
- No solo para storage, para TODOS resources
- Descubre dependencies
- Migra VMs, databases, aplicaciones
- Integra Data Box para bulk transfers
- Perfecto para: Full datacenter migration

### 8. Premium Performance Options (Performance Premium)

#### a) Tipos Premium

**1. Premium Block Blobs**
- Storage type: Blob
- Use case: AI, IoT analytics (small files, low latency needed)
- Redundancia: LRS o ZRS SOLO (no geo)

**2. Premium Page Blobs**
- Storage type: Page Blob (unmanaged VM disks)
- Use case: Virtual machine disks
- Redundancia: LRS SOLO (más restrictivo)

**3. Premium File Shares**
- Storage type: Azure Files
- Use case: High-performance enterprise apps
- Supports: SMB (Windows) + NFS (Linux)
- Redundancia: LRS o ZRS SOLO

#### b) Trade-off
- **Más performance** = **Menos redundancia**
- Premium types = Single region redundancy only
- Para máxima HA, sacrificas performance

## Comparación Storage Types

| Tipo | Contenido | Estructura | Acceso | Tier | Casos |
|------|----------|-----------|--------|------|-------|
| **Blob** | Cualquier archivo | Container/Blob | URL pública | Hot/Cool/Archive | Media, CDN |

## 📚 Conclusión y Síntesis Profunda - Capítulo 06: Storage

### ¿Qué Aprendiste Realmente?

**Storage es donde VIVE tu data.**

Si Compute es "dónde ejecutas tu código," Storage es "dónde almacenas el resultado(y todo lo demás)."

El capítulo cubrió dos preguntas críticas:
1. **QUÉ tipo de storage** (Blob, Disk, Files, Archive)
2. **CÓMO protejo mi data** (Redundancia: LRS, ZRS, GRS, GZRS)

### Storage Accounts: Tu Namespace

Todo storage en Azure vive en un Storage Account. Es tu "contenedor de almacenamiento."

**Importante:** Nombre debe ser globalmente único. No puedes tener dos "mystorageaccount" en Azure—el primero la reclama.

```
Store Account: "mycompany"
└─ Web address: mycompany.blob.core.windows.net
└─ URL única globalmente: https://mycompany.blob.core.windows.net/container/file.jpg
```

### Los 4 Tipos de Storage: Decidir QUÉ

**Blob Storage: "Archivos Flexibles"**

```
Uso: Imágenes, videos, documentos, logs, backups
Tamaño: Desde bytes hasta terabytes
Estructura: Storage Account → Container → Blob

3 tipos de Blobs:
├─ Block Blobs (80% de uso): Streaming, logs, backups
├─ Append Blobs (15% de uso): Logging (append-only)
└─ Page Blobs (5% de uso): VHD disks, random access

Más barato por byte, pero acceso relativamente lento.
```

**Disk Storage: "VHD de VM"**

```
Uso: Discos para VMs
No piensas en esto directamente—Azure lo maneja cuando createas VM.
Pero importante saber que existen diferentes tipos de discos:
├─ HDD (lento, barato): Testing, backups
├─ Standard SSD (rápido, medio): Producción típica
├─ Premium SSD (muy rápido, caro): Databases, high-performance
└─ Ultra SSD (ultra-rápido, muy caro): Enterprise, gaming

Managed = Azure administra físicamente, garantiza disponibilidad.
```

**File Storage: "Compartir Archivos"**

```
Uso: Network shared drives (como NAS)
Protocolo: SMB (Windows) o NFS (Linux)

Beneficio: Reemplaza on-premises file servers
└─ Escalable (unlimited)
└─ Automatizado (backups de Azure)
└─ Accessible (desde anywhere con VPN/ExpressRoute)

Vs alternatives:
└─ Blob: Flexible pero sin "file system" nativo
└─ File: Verdadero file system compartido
```

**Archive Storage: "Data Antigua y Barata"**

```
Uso: Compliance archival, backups históricos, cold storage
Costo: ¨¨¨ muy bajo (cents per TB per month)
Acceso: Muy lento (esperas horas para recuperar)

Caso: "Debo guardar logs de 2 años por compliance,
      pero duredo los voy a necesitar"
      
→ Archive Storage = $1 al mes mover todo ahí
→ Si necesito: "Espero 15 horas, restored, ahora puedo acceder"
```

### Tiers de Pricing: Costo vs Acceso

Aquí es donde mucha gente pierde dinero:

```
                  | Storage Cost | Access Cost | Latency
────────────────────────────────────────────────────────────
Hot      (frecuente) | $$$ | ¢ | Rápido
Cool     (infrequente) | $$ | $$ | Más lento
Archive  (nunca) | $ | $$$+ | Muy lento

Decisión:
¿Accedes constante?
  └─ Hot (aunque caro, acceso es barato)
  
¿Accedes ocacionalmente?
  └─ Cool (storage barato, acceso caro se balancean)
  
¿Nunca accedes?
  └─ Archive (mucho storage barato)
```

**Trampa común:**

```
"Guardaré archivos en Hot Tier para ahorrar acceso"
→ Pagas $123/mes en storage para ahorrar $2 en acceso ❌

Correcto: "Estos archivos no se acceden, subo a Archive"
→ Pagas $1/mes, data segura, cumplo compliance ✅
```

### Redundancia: Cómo Protejes Tu Data (¡CRÍTICO!)

Aquí es donde muchos no entienden límites; **Este es el concepto más importante del capítulo.**

Data falla. Discos se corrompen. Datacenters exploran. Necesitas protección.

Azure automáticamente replica tu data. **Mínimo 3 copias siempre.** La pregunta es: ¿DÓNDE esas copias?

```
┌────────────────────────────────────────────────────────────┐
│ LRS (Locally Redundant Storage)                            │
│ 3 copias en MISMO datacenter/zone                          │
│ ✅ Protege: Disk failure                                    │
│ ❌ NO protege: Zona outage, región outage                   │
│ $ Costo: Bajo                                               │
│ Cuándo use: Testing, no-critical (puedo re-crear)          │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│ ZRS (Zone Redundant Storage)                               │
│ 3 copias en 3 DIFERENTES ZONES (misma región)             │
│ ✅ Protege: Zone outage                                     │
│ ❌ NO protege: Región outage completo                       │
│ $$ Costo: Medio                                            │
│ Cuándo use: Balance entre costo y HA verdadera            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│ GRS (Geo-Redundant Storage)                                │
│ 3 copias región primaria (LRS) + 3 copias región paired    │
│ ✅ Protege: Región outage                                   │
│ ❌ Limitación: Copias primarias en MISMO zone (no ZRS)     │
│ $$$ Costo: Alto (10x LRS)                                  │
│ Cuándo use: Disaster recovery crítico                      │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│ GZRS (Geo-Zone-Redundant Storage)                          │
│ 3 copias spans 3 zones primaria + 3 en región paired      │
│ ✅ Protege: Zone outage + Región outage                     │
│ $$$$$ Costo: Muy alto (20x LRS)                            │
│ Cuándo use: Data CRÍTICA que NUNCA puede fallar            │
└────────────────────────────────────────────────────────────┘
```

**Decisión matriz:**

```
¿Qué tan importante es tu data?

"Es un cache que puedo re-crear"
  → LRS (barato)

"Es data importante pero si falla, puedo restaurar desde otro lado"
  → ZRS (HA zona-local)

"Es data crítica que debe sobrevivir región outage"
  → GRS (caro pero protegido)

"Es data ULTRA CRÍTICA que NUNCA puede desaparecer"
  → GZRS (muy caro pero máxima protección)
```

### Data Migration: Mover Data HACIA/DESDE Azure

Espacio en disco lleno on-premises? Necesitas mover a cloud.

**Para pequeñas cantidades:**
```
AzCopy (command line): Perfecto para scripts/automatización
Storage Explorer (GUI): Fácil para humanos, drag & drop
```

**Para GRANDES cantidades:**
```
Azure Data Box (1 PB+ data)
└─ Microsoft envía disco físico encrypted
└─ Copias tu data al disco
└─ Lo envías de vuelta a Azure
└─ Azure importa automáticamente
   
Por qué: Transferir 1 PB por internet = "jajaja no, gracias"
Vs. "Física delivery en 2 semanas"
```

### Caso Práctico Completo

Eres architect de TechCorp:

**Requisito 1:** "Tengo 500 GB de photos de usuario. Acceso frecuente."
**Decisión:** 
- Blob Storage
- Hot Tier (acceso frecuente)
- ZRS redundancy (protege zone failures)

**Requisito 2:** "Logs de hace 2 años. Compliance dice guardar 7 años. Nunca los acceso."
**Decisión:**
- Blob Storage
- Archive Tier (muy barato)
- LRS redundancy (no los necesito, si se pierden re-generaré)

**Requisito 3:** "Desplego VMs. Necesitan disks rápidos."
**Decisión:**
- Disk Storage
- Premium SSD (high-speed es crítico)
- Azure maneja red.

**Requisito 4:** "Equipos remotos necesitan acceder files compartidos."
**Decisión:**
- File Storage
- Estándar tier
- ZRS (debe estar disponible siempre)

**Resultado arquitectura:**
```
Storage Account "TechCorp-Data"
├─ Blob Container "UserPhotos" → Hot, ZRS (frequente access, important)
├─ Blob Container "Archive Logs" → Archive, LRS (nunca acceso)
├─ Disk Storage "VM-Disks" → Premium SSD, Azure managed
└─ File Storage "Team Drives" → Standard tier, ZRS
```

### Conexión con Todo

- **Capítulo 4 (Compute):** VMs necesitan disks (Disk Storage)
- **Capítulo 5 (Networking):** Data viaja por red
- **Capítulo 11 (Pricing):** Storage es costa, planifica bien

### Lo Que Deberías Saber AHORA

1. "¿Dónde almaceno logs que genero constantemente?"
   **Respuesta:** Blob Storage, Hot Tier, Block Blobs, ZRS

2. "¿Cómo protejo data que NUNCA puede perderse?"
   **Respuesta:** GZRS (o GRS si región outage no es preocupación)

3. "¿Cuándo uso Archive?"
   **Respuesta:** Data que es relativamente imposible que necesites

4. "¿Blob vs Disk vs Files?"
   **Respuesta:** Blob (flexible), Disk (VMs), Files (network shares)

Como organismo vivo, data es memoria. Sin buena storage, técnicamente aplicación funciona pero PERDERÁS data.

Storage bien diseñado = Data segura, costo optimizado, cumplimiento.
