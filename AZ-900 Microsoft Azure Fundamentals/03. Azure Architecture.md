# 03. Azure Architecture (Arquitectura de Azure)

## Resumen del Capítulo
Este capítulo explora los componentes fundamentales de la arquitectura de Azure: regiones, zonas de disponibilidad, grupos de recursos y Azure Resource Manager. Estos elementos son esenciales para entender cómo está construida la plataforma Azure.

## Temas Principales

### 1. Regiones de Azure (Regions)

#### a) Definición
- **Región**: Conjunto de datacenters desplegados dentro de un perímetro definido por latencia, conectados a través de red regional de baja latencia
- Paráfrasis: 2+ datacenters físicos, no muy alejados, conectados por fibra

#### b) Componentes de una Región
- **Set de Datacenters**: Cada región tiene múltiples datacenters
- **Latencia Definida**: Los datacenters NO están muy lejos (típicamente <10ms entre ellos)
- **Red de Baja Latencia**: Conexión por fibra óptica (más rápida disponible)

#### c) Criterios para Elegir una Región
1. **Localización Geográfica**: Elige región cercana a usuarios
   - Objetivo: Minimizar latencia para usuarios
   
2. **Disponibilidad de Características**: 
   - No todas las features en todas las regiones
   - Regiones nuevas tienen menos servicios
   - Si necesitas característica específica, tienes menos opciones
   
3. **Precios Diferentes**:
   - Pueden variar 20-30% entre regiones
   - A veces no puedes tener precio, latencia Y features

#### d) Ejemplos de Regiones
- East US (Virginia)
- West US
- Europe North
- Southeast Asia
- Australia East
- Brazil South
- etc.

### 2. Paired Regions (Regiones Emparejadas)

#### a) Concepto
- Cada región está emparejada con OTRA región
- Pares generalmente en misma área geográfica
- EXCEPCIÓN: Brazil South emparejado con South Central US

#### b) Ejemplos de Pares
- East US ↔ West US
- France Central ↔ France South
- Australia East ↔ Australia Southeast
- North Europe ↔ West Europe

#### c) Beneficios
- **Multi-region Failover**: Si región primaria cae, puedes cambiar a secundaria
- **Prioridad de Recuperación**: En outage multi-región, se prioriza al menos una región del par
- **Actualizaciones Serializadas**: Azure actualiza solo UNA región del par a la vez (mantenimiento planificado)
- **Replicación Automática**: Algunos servicios replican automáticamente (geo-redundant storage)

### 3. Availability Zones (Zonas de Disponibilidad)

#### a) Definición
- **Availability Zone**: Ubicación física única dentro de una región de Azure
- Compuesta de 1+ datacenters propios
- Cada zona tiene: potencia independiente, refrigeración, red

#### b) Características
- **Mínimo 3 zonas** por región (para las que las soportan)
- No todas las regiones tienen Availability Zones
- Separación física protege contra fallos de datacenters

#### c) Analogía LEGO
- Tabla = Región
- Lego Bricks 1,2,3 = 3 Availability Zones
- Cada uno con potencia, refrigeración, red propios
- Si se cae un brick, otros 2 siguen sirviendo

#### d) Uso para Alta Disponibilidad
- Distribuye VMs entre múltiples zones
- Si zone 1 cae, zones 2 y 3 siguen
- IMPORTANTE: Necesitas desplegar en múltiples zones para beneficiarse
- Una VM en una zone NO protege de outage de zone

#### e) Zone-Redundant Services
- Cloud Storage automáticamente replica entre zones
- Se replica sin que hagas nada manual
- Ejemplo: Si agregas storage, se agrega automáticamente a las otras zonas también

### 4. Relación: Región vs Availability Zone
```
REGIÓN (Area Geográfica Grande)
├─ ZONA 1 (Datacenters físicos)
│  ├─ Potencia independiente
│  ├─ Refrigeración independiente
│  └─ Red independiente
├─ ZONA 2 (Datacenters físicos)
│  └─ (Independiente de Zona 1)
└─ ZONA 3 (Datacenters físicos)
   └─ (Independiente de Zonas 1 y 2)
```

### 5. Resource Groups (Grupos de Recursos)

#### a) Definición
- **Container lógico** que contiene recursos relacionados
- Cada recurso en Azure DEBE estar en un resource group
- El resource group EN SÍ NO es un recurso

#### b) Características Clave
- **Un recurso = Un resource group**: Cada recurso solo puede estar en un resource group
- **Flexibilidad**: Tú decides cómo organizar recursos en groups
- **Múltiples regiones**: Un resource group puede contener recursos de diferentes regiones
- **Etiquetado (Tagging)**: Asigna labels a recursos para organización y facturación
- **Control de Acceso**: Otorga permisos a todo un resource group

#### c) Beneficios
- Organización lógica
- Gestión agrupada de permisos
- Facilita provisioning/desprovisioning juntos (elimina grupo = elimina todos los recursos)
- Claridad en facturación (por tags)

### 6. Azure Resource Manager (ARM)

#### a) Definición
- **La puerta de entrada** para todo en Azure
- Servicio CENTRAL de deployment y management
- Todos los accesos pasan por ARM (Portal, CLI, PowerShell, APIs, SDKs)

#### b) Características
- **Single Entry Point**: Garantiza consistencia
- **API Única**: CLi, Portal, PowerShell, todo va a través de ARM

#### c) Beneficios
- **Deployment Consistente**: Cada vez se despliega igual
- **Gestión de Dependencias**: Define relaciones entre recursos
- **Role-Based Access Control**: Control fino de permisos
- **Tagging and Organization**: Etiquetado lógico
- **Billing Clarity**: Visualiza costos por tags

#### d) ARM en la Práctica
- Cuando activas Portal → ARM maneja tu request
- Cuando usas CLI → ARM procesa el comando
- Cuando corres PowerShell → ARM ejecuta el cmdlet
- Todo es consistente porque todo pasa por ARM

### 7. Resumen de Estructura Azure

```
SUSCRIPCIÓN (Billing Entity)
└─ TENANT (Azure AD Instance - Tu Organización)
   └─ REGIÓN (Ubicación Geográfica)
      ├─ AVAILABILITY ZONE 1
      ├─ AVAILABILITY ZONE 2
      └─ AVAILABILITY ZONE 3
         └─ RESOURCE GROUP (Container Lógico)
            ├─ Recurso 1 (VM, Storage, etc)
            ├─ Recurso 2
            └─ Recurso 3 (Todo pasa por ARM)
```

## Puntos Clave para el Examen

✅ **Recuerda**:
- **Región**: 2+ datacenters en área geografica, conectados por fibra
- **Regiones Emparejadas**: Para multi-region failover y mantenimiento
- **Availability Zones**: 3+ zonas de fallo independientes dentro de región
- **Resource Group**: Container LÓGICO, no físico
- **ARM**: Centro de todo en Azure, consistency garantizado
- Cada recurso en un SOLO resource group
- El resource group necesita ubicación (metadata)
- Ningún servicio = Sin resource group

| Término | Definición |
|---------|-----------|
| **Latencia** | Tiempo viaje de datos |

## 📚 Conclusión y Síntesis Profunda - Capítulo 03: Azure Architecture

### ¿Qué Aprendiste Realmente?

Este capítulo responde la pregunta fundamental: **"¿DÓNDE pongo mis cosas en Azure?"**

No es suficiente saber que puedes crear una VM. Necesitas saber **dónde crearla, en qué región, con qué redundancia, en qué grupo de recursos**, para que tu aplicación sea rápida para usuarios, segura si fallo, y organizada para que tu equipo no se pierda.

El capítulo cubrió three conceptos absolutamente fundamentales que se repiten EN TODO lo que hagas en Azure:

1. **Regiones** - Ubicación física (Australia, Europa, USA)
2. **Availability Zones** - Sub-ubicaciones dentro de región (para HA)
3. **Resource Groups** - Contenedores lógicos (para organización)

### Las Regiones: "Donde Vive tu Data"

Aquí hay un hecho que sorprende a muchos: **La región que eliges afecta TODO.**

```
Si eres una appdev en Argentina:
Opción A: "Voy a usar las VMs en USA porque es más barato"
Resultado: Usuarios en Argentina haciendo request = Viaja a USA = 150ms latencia
           = Aplicación lenta = Usuarios irados

Opción B: "Voy a usar las VMs en South America (Brazil)"
Resultado: Usuarios en Argentina haciendo request = Viaja a Brazil = 50ms latencia
           = Aplicación rápida = Usuarios felices
```

**La latencia IMPORTA.** La gente no espera. Sin embargo, a veces tienes que balancear:

```
                    LATENCIA BAJA    FEATURES    PRECIO
East US             ✅               ✅           ✅
Brazil South        ❌ (lejos)       ⚠️           ✅
Europe North        ❌ (lejos)       ❌           ✅
```

Ves? **No puedes tenerlo todo.** Así que:

- Startups pobres: Elige precio + features. La latencia es sacrificio.
- Aplicaciones críticas: Elige latencia baja + disponibilidad alta. Pagas más.
- Aplicaciones reguladas: Elige región que cumpla compliance (si datos sensibles deben estar en EU, no hay opción—es EU)

### Regiones Emparejadas: "Tu Plan B"

Las regiones emparejadas es un concepto subestimado pero CRÍTICO.

Microsoft sabe que datacenters fallan. Ciudades tienen terremotos. Operadores cliquean botones equivocados y ponen datacenters offline.

Entonces, durante mantenimiento planificado:
- Azure está actualizando East US (la región)
- ¿Qué hace Azure? **Apaga todos los servidores de actualización en East US, pero deja West US intacto**

¿Por qué? Porque son emparejadas. Microsoft se asegura que SIEMPRE haya al menos una región del par funcionando. 

En caso de desastre:
- East US explota (literalmente, un asteroide)
- West US sigue funcionando
- Tu data se replicó automáticamente a West US (si usas Geo-Redundant Storage)
- Tu app no muere

**Eso es el poder de paired regions.**

Memoriza: No es una feature extra. **Es el motor del Disaster Recovery de Azure.**

### Availability Zones: "Tolerar Fallos Locales"

Aquí es donde muchos confunden. AZ (Availability Zones) NO es lo mismo que regiones emparejadas.

```
Regiones Emparejadas: "Si toda la región de USA cae, el Brasil tiene backup"
Availability Zones: "Si el datacenter 1 en USA cae, datacenters 2 y 3 en USA siguen funcionan"
```

Es el nivel DEBAJO de región.

**Caso práctico:**

Tienes 3 VMs:
- VM1 en Zone 1
- VM2 en Zone 2
- VM3 en Zone 3

El datacenter del Zone 1 tiene apagón eléctrico (rayos, problema, lo que sea).

**Sin AZ awareness:**
```
VM1: ❌ OFFLINE
VM2: ✅ online
VM3: ✅ online
```

Resultado: 33% de tu capacidad desaparece. Si tráfico sube, los otros 2 servidores se saturan.

**Con AZ awareness:**
```
Load Balancer detecta VM1 offline
Automáticamente redirige TODO tráfico a VM2 + VM3
Usuarios no notan nada
Azure reparó falla y VM1 vuelve online
```

**Eso es alta disponibilidad verdadera.**

La lección: Si despliegas una sola VM en una sola Zone, **no tienes verdadera HA.** Necesitas múltiples VMs en múltiples Zones.

### Resource Groups: "Tu Estructura Organizacional"

Este es un concepto que suena simple pero confunde a muchos.

Un Resource Group **NO es un datacenter.** NO es una red. **NO es nada físico.**

Es un **contenedor lógico.**

```
// INCORRECTO:
"Mi database en Resource Group X"  ← Confunde location

// CORRECTO:
"Mi database (que está en Region East US, Zone 1) 
 está organizado en Resource Group "Production-Apps""
```

Entonces, ¿para qué sirve un resource group?

**Organización:**
```
Empresa "TechCorp"
├─ Resource Group "WebApp-Production"
│  ├─ VM web-server-1
│  ├─ VM web-server-2
│  ├─ Load Balancer
│  └─ Storage Account
│
└─ Resource Group "Database-Production"
   ├─ Azure SQL Database
   ├─ Backup storage
   └─ Managed Disk
```

**Facturación:**
```
Con tagging:
└─ Resource Group "WebApp-Production"
   ├─ VM web-server-1 (tag: environment=prod, team=frontend)
   ├─ VM web-server-2 (tag: environment=prod, team=frontend)
```

Luego, Azure dice: "El equipo de frontend pagó $3,000 en compute este mes"

**Permisos:**
```
Manager de proyecto: "Permiso al resource group WebApp-Production"
Resultado: Puede hacer TODO dentro ese grupo (crear/modificar/eliminar)
           Pero NO puede tocar Database-Production
```

**Eliminación en una línea:**
```powershell
az group delete --name "WebApp-Production"
```

Eso elimina TODOS los recursos dentro. Es como "eliminar carpeta" en Windows—desaparecen todos los archivos adentro.

**Lección:** Resource Groups es cómo organizas el caos de Azure. Sin ellos, tienes 500 recursos sin estructura. Con ellos, tienes orden.

### Azure Resource Manager (ARM): "El Director de Orquesta"

ARM NO es ARM Templates. Eso confunde a muchos.

**ARM (Azure Resource Manager) = Servicio central de Azure**

Cada acción en Azure pasa por ARM:
1. Haces click en Portal → Portal habla a ARM
2. Usas CLI → CLI habla a ARM
3. Despliegas con PowerShell → PowerShell habla a ARM
4. Escribes ARM Template → Template habla a ARM

ARM es el **árbitro** que dice "sí, puedo crear esto" o "no, violas policy".

**Beneficios:**

1. **Consistencia:** 
   - Despliegas mediante Portal
   - Despliegas mediante CLI
   - Resultado: Exactamente igual

2. **Dependencias:**
   ARM entiende relaciones. "No crearás VM sin Subnet. No replicarás Storage sin Region." Previene errores.

3. **Role-Based Access Control:**
   ARM verifica permisos. "¿Puedo crear VM? Déjame verificar en AAD..."

4. **Tagging y Organización:**
   ARM administra metadata.

### La Jerarquía Completa: Verlo Todo Junto

Ahora que entiendes regiones, AZ, resource groups, ARM, aquí está cómo se conecta:

```
SUSCRIPCIÓN (Billing, Contrato)
│
├─ TENANT (Azure AD, Identidad)
│  │
│  └─ REGIÓN: East US
│     │
│     ├─ Availability Zone 1
│     │  └─ Resource Group: WebApp
│     │     └─ VM web-1 (Aquí está deployment REAL)
│     │
│     ├─ Availability Zone 2
│     │  └─ Resource Group: WebApp
│     │     └─ VM web-2 (Backup automático aquí)
│     │
│     └─ Availability Zone 3
│        └─ Resource Group: WebApp
│           └─ VM web-3 (Extra capacity aquí)
│
├─ REGIÓN: West US (Paired Region)
│  │
│  └─ [Geo-replicated backup si desastre]
```

**TODO pasa por ARM.** ARM decide si cada acción es legal, segura, cumple policies.

### Caso de Uso Práctico Completo

Imagina eres architect de "MegaApp":

**Requisito 1:** "Usuarios en Europa necesitan rápido acceso"
**Decisión:** Usa Europe North region

**Requisito 2:** "Si Zone 1 explota, app sigue corriendo"
**Decisión:** Despliega en Zones 1, 2, 3. Load Balancer entre ellos

**Requisito 3:** "Team de Frontend, Backend, Database son equipos separados. Cada uno maneja su propio presupuesto"
**Decisión:** 
- Resource Group "Frontend-Prod"
- Resource Group "Backend-Prod"  
- Resource Group "Database-Prod"

**Requisito 4:** "Compliance europea: Data NO puede salir de Europa"
**Decisión:** Paired region es Europe West (no importa, igual en Europa). Si disaster, falla over a otro datacenter European.

**Resultado:** Tu arquitectura es:
- Rápida (usuarios acceden localmente en Europe)
- Resiliente (3 Zones protegen contra fallos locales)
- Altamente disponible (si Zone cae, load balancer redirige)
- Organizada (3 resource groups, 3 equipos)
- Compliant (todo en Europa)
- Escalable (agregar más VMs es un template JSON)

**Pensar en arquitectura así es lo que hace difference entre amateur y profesional.**
