# 10. Security (Seguridad)

## Resumen del Capítulo
Security es crítica en Azure. Este capítulo cubre defensa en profundidad (Defense in Depth), Azure Security Center, protección de red, encriptación, y mejores prácticas de seguridad. Este capítulo **SÍ es parte** del examen AZ-900.

## Temas Principales

### 1. Defense in Depth (Defensa en Profundidad)

#### a) Concepto
- **Múltiples capas de seguridad**
- No confiar en una sola línea de defensa
- Si attacker bypasses 1 capa, quedan otras

#### b) Capas de Seguridad (De fuera hacia dentro)

```
EXTERIOR
    ↓
1. PHYSICAL SECURITY (Datacenters)
   - Físicamente protegido
   - Acceso limitado
   
2. PERIMETER SECURITY (DDoS protection)
   - Azure DDoS Protection
   - Bloquea ataques masivos
   
3. NETWORK SECURITY (Firewalls, NSGs)
   - Network Security Groups
   - Azure Firewall
   - VPN/ExpressRoute
   
4. APPLICATION SECURITY
   - Input validation
   - Web Application Firewall
   - Secure coding
   
5. DATA SECURITY (Encryption)
   - Encryption at rest
   - Encryption in transit
   - Key Management (Key Vault)
   
6. IDENTITY & ACCESS
   - Azure AD
   - MFA
   - RBAC
   
7. LOGGING & MONITORING
   - Azure Monitor
   - Security Center
   - Audit logs
   
INTERIOR (Tu responsabilidad)
```

#### c) Beneficio
- Attacker necesita romper MÚLTIPLES capas
- Complexity = Deterrent

### 2. Network Security Groups (NSGs)

#### a) Definición
- **Firewall virtual** para subnets y NICs
- Controla incoming/outgoing traffic
- Basado en reglas

#### b) Reglas NSG
```
Inbound Rules:
  - Quién puede ENTRAR a la subnet/VM
  - Basado en: Source IP, port, protocol
  
Outbound Rules:
  - Quién puede SALIR de la subnet/VM
  - Similar a inbound
```

#### c) Ejemplos de Reglas
```
Allow HTTP incoming on port 80
  → Usuarios pueden acceder website

Allow SSH from 10.0.0.0/24 only
  → Solo computadoras en subnet específica

Deny ALL outbound except DNS
  → VM solo puede hablar a DNS servers
```

### 3. Azure Firewall

#### a) Definición
- **Managed firewall service**
- Completa protección de red
- Basada en Cloud

#### b) Características
- Stateful firewall (mantiene estado)
- High availability integrada
- Logging y monitoring
- Protección DDoS básica

#### c) Diferencia: NSG vs Firewall
```
NSG:
  - Layer 4 (Transport)
  - Subnet/NIC level
  - Más granular

Azure Firewall:
  - Layer 7 (Application)
  - Network level
  - Inspección profunda
```

### 4. DDoS Protection

#### a) Qué es DDoS
- **Distributed Denial of Service**
- Ataques intentan sobrecargar servicio
- Millones de requests simultáneamente
- Servicio cae

#### b) Azure DDoS Protection
- **Standard**: Básico, siempre activo
- **Premium**: Avanzado, detection + mitigation
- Protege aplicaciones automáticamente

### 5. Azure Security Center

#### a) Definición
- **Central security dashboard** para Azure
- Monitorear seguridad de todos tus resources
- Recomendaciones automáticas
- Compliance tracking

#### b) Funcionalidades
- **Threat Detection**: Detecta ataques potenciales
- **Vulnerability Assessment**: Identifica debilidades
- **Recommendations**: Acciones para mejorar
- **Incidents**: Alertas en tiempo real

#### c) Scoring de Seguridad
- Calificación 0-100
- Basado en implementación de mejores prácticas
- Tareas específicas para mejorar score
- Benchmarking contra industry standards

### 6. Encryption (Encriptación)

#### a) Tipos de Encriptación

**1. Encryption at Rest**
- Data almacenado está encriptada
- En disco, en database
- Clave necesaria para leer
- Default en muchos servicios Azure

**2. Encryption in Transit**
- Data en movimiento está encriptada
- Entre client y server
- SSL/TLS típicamente
- HTTPS vs HTTP
- Protege contra interception

**3. End-to-End Encryption**
- Encriptada desde origen hasta destino
- Incluso Azure no puede leer
- Usuario controla claves

#### b) Azure Key Vault
- **Manages encryption keys**
- Almacena secrets, passwords, certificates
- Access control integrado
- Auditing completo
- Integración con otros servicios

### 7. Azure Information Protection

#### a) Definición
- **Data classification + protection**
- Categoriza qué tan sensible es data
- Aplica protección automáticamente

#### b) Etiquetas de Clasificación
```
Public:   Información pública, sin restricción
Internal: Solo para empleados
Confidential: Sensible, acceso restringido
Secret:   Highest sensitivity, muy pocas personas
```

#### c) Protecciones
- Encriptación automática
- Watermarks visuales
- Restricciones de acceso
- Expiration automática

### 8. Compliance y Governance

#### a) Azure Policy
- **Enforce compliance** en todo Azure
- Define reglas y estándares
- Automáticamente aplica
- Previene configuraciones inseguras

#### b) Ejemplos de Policies
```
"VMs deben tener MFA"
"Data debe estar encrypted"
"Solo ciertos regions permitidos"
"Tags requeridos en todos resources"
```

#### c) Azure Blueprints
- **Pre-defined environments** completos
- Implementan compliance automáticamente
- Reutilizable
- Versionado

### 9. Advanced Threat Protection

#### a) Azure Advanced Threat Protection (ATP)
- Detecta ataques sofisticados
- Basado en machine learning
- Identifica patrones anormales
- Alertas in real-time

#### b) Application Security (ASC)
- Application Gateway Web Application Firewall
- Protección contra: XSS, SQL injection, etc
- Layer 7 protection

### 10. Auditing y Compliance

#### a) Azure Audit Logs
- Registro de ACCIÓN en Azure
- "Quién hizo QUÉ cuándo"
- Compliance requirement
- Retenido por 90 días (extendible)

#### b) Azure Monitor
- Monitoring y alerting completo
- Logs de aplicaciones
- Performance metrics
- Behavior analysis

### 12. Best Practices Security

✅ **SIEMPRE**:
1. Enable MFA (Multi-Factor Authentication)
2. Use strong passwords o passwordless
3. Encrypt data (at rest + in transit)
4. Monitor y audit everything
5. Principle of Least Privilege
6. Keep software updated
7. Regular backups
8. Defense in Depth
9. Security awareness training
10. Regular security reviews

❌ **NUNCA**:
1. Share credentials
2. Store passwords en plain text
3. Disable security features
4. Use default passwords
5. Trust only perimeter
6. Ignore security alerts
7. Store secrets en code

## 📚 Conclusión y Síntesis Profunda - Capítulo 10: Security

### ¿Qué Aprendiste Realmente?

**Security NO es una característica. Es una filosofía.**

Si Identity es "quién accede," Security es "cómo protegemos lo que acciden."

El capítulo no fue "click para activar security." Fue "Security es múltiples capas, cada una cumple función."

### Defense in Depth: El Concepto Unificador

```
Arquitecto antiguo:
├─ "Pongamos firewall"
├─ "Problema resuelto"
└─ ❌ No sabe que CUALQUIER firewall se bypasea

Arquitecto moderno:
├─ Entiende que NINGUNA defensa es perfecta
├─ Agrega MÚLTIPLES defensas
├─ Attacker necesita romper TODAS
└─ Complejidad = Deterrent
```

**Las 7 capas (fuera a dentro):**

```
1. PHYSICAL: Datacenters cerrados, acceso limitado
2. PERIMETER: DDoS protection, bloquea ataques masivos
3. NETWORK: Firewalls, NSGs, bloquea traffic no-autorizado
4. APPLICATION: Input validation, no executa código malicioso
5. DATA: Encryption, data no leible si stolen
6. IDENTITY: AAD, MFA, verifica quién eres
7. LOGGING: Auditoría, detecta anomalías
```

Attacker con herramientas penetrando layer 1? Layers 2-7 lo detectan.

### Network Security Groups: "Firewall Virtual"

NSG es lo que controlas DIRECTAMENTE en security.

```
NSG Rules:

Inbound: "Quién entra"
├─ Allow HTTP on port 80 (website)
├─ Allow HTTPS on port 443 (website secure)
├─ Deny SSH from internet
└─ Allow SSH desde 10.0.0.0/24 only (internal)

Outbound: "Quién sale"
├─ Allow DNS (para resolver domains)
├─ Allow HTTPS (para updates)
├─ Deny everything else (paranoia)
```

**Por qué paranoia?** Porque VM comprometida = Trata communicarse C&C server. Si Deny outbound por defecto, "communication bloqueado."

### DDoS Protection: "Against Massive Attacks"

DDoS = Millions attacks simultáneously.

```
Normal attack: 1000 requests/sec
DDoS: 100 million requests/sec

Tu VM:
├─ "Puedo handle 10,000 requests/sec"
├─ Receive 100 million
└─ _se cae_

Azure DDoS Protection:
├─ Detecta patrón attack
├─ Bloquea malicious traffic
├─ Legitimo traffic = allowed
└─ Tu app keeps running
```

**Tres capas DDoS:**

```
Standard: Automático, siempre on, básico
└─ Incluido free

Premium: Advanced, detection + mitigation, $$$
└─ Para sitios public critical (banco, carrera política, etc)

App Gateway: Layer 7 protection contra web attacks
└─ SQL injection, XSS, etc
```

### Azure Security Center: "Dashboard Central"

Security Center es tu "security control panel."

```
Sin Security Center:
├─ 100 VMs
├─ ¿Cuál tiene antimalware?
├─ ¿Cuál tiene updates?
├─ ¿Cuál es vulnerable?
└─ No idea 💥

Con Security Center:
├─ Ves todos 100 VMs
├─ "Red flag" en los problemáticos
├─ Recomendaciones automáticas
└─ Actualiza todo de una vez
```

**Scoring:**

```
Security Score 0-100
├─ 100 = Perfectamente securizado
├─ 50 = Average
└─ 0 = ¡Ayuda!

Azure dice: "Si implementas recomendación X, score sube de 50 a 65"
```

Gamified security. Funciona.

### Encryption: "Data Incomprehensible"

```
Sin encryption:
└─ Attacker robo harddrive
   └─ Ve todos tus passwords
   └─ Accede todo

Con encryption:
└─ Attacker robo harddrive
   └─ Ve gibberish
   └─ Sin key, inutilizable
```

**Dos tipos:**

```
Encryption at Rest:
├─ Data almacenado encriptado
├─ En disco, database, storage
├─ Automaticamente en Azure (muchos servicios)
└─ Por qué: Si harddrive stolen, data protected

Encryption in Transit:
├─ Data en movimiento encriptado
├─ Entre devices
├─ HTTPS (website), SSL/TLS (connections)
└─ Por qué: Si packet intercepted mid-travel, unreadable
```

### Key Vault: "Guardia de Secretos"

```
Mal:
├─ Hardcodeo password en app
├─ Git commit accidental (oops)
├─ Code leak = Todos ven password
└─ 💀

Bien (Key Vault):
├─ Password en Key Vault
├─ App pide "dame password"
├─ Key Vault: "Verifico quién eres"
├─ Solo tú (trusted app) lo tienes
└─ 🔐
```

Key Vault guarda:
```
├─ Encryption keys (para encrypt/decrypt)
├─ Secrets (passwords, connection strings)
├─ Certificates (SSL certs)
└─ Todo con audit trail (quién accesó qué, cuándo)
```

### Compliance y Azure Policy

```
Compliance = "Cumplir regulaciones"

Ejemplo: HIPAA (healthcare regulation)
└─ "Data debe estar encrypted"
└─ "Data nunca debe salir USA"
└─ "Audit logs 7 años"

Cómo enforcer sin Azure Policy:
├─ Manually chequear cada VM
├─ Humano: "Oops, olvide encryption en VM 47"
└─ Compliance failed, regulador angry

Con Azure Policy:
├─ "ALL VMs MUST have encryption"
├─ Policy automáticamente applica
├─ Violation? VM creation blocked
└─ Compliance guaranteed
```

### Shared Responsibility

```
ON-PREMISES: TODO TÚ

IaaS: COMPARTIDO
├─ TÚ: OS, aplicaciones, data
├─ Azure: Hardware, red

PaaS: MÁS AZURE

SaaS: AlMOST TODO AZURE
└─ TÚ: Data, dispositivos
```

**La lección crítica:** Mientras MÁS managed, menos TÚ responsable. Peroooooo también MENOS control.

### Caso de Uso: Architecturing Seguridad

TechCorp aplicación:

**Requisito 1:** "Data debe estar encrypted"
**Implementación:**
```
├─ Encryption at rest: Azure handles (default)
├─ Encryption in transit: HTTPS only (NSG permite 443, deniega 80)
└─ Keys en Key Vault (no en code)
```

**Requisito 2:** "Detectable si data va missing"
**Implementación:**
```
├─ Audit logs (quién accesó qué)
├─ Security Center monitoring
├─ Alertas si anomalía
└─ "3am, 100 downloads? Alert!"
```

**Requisito 3:** "Protegido contra DDoS"
**Implementación:**
```
├─ Application Gateway (layer 7 protection)
├─ DDoS Protection Premium
└─ "Attack: automáticamente mitigated"
```

**Requisito 4:** "Compliance: GDPR"
**Implementación:**
```
├─ Azure Policy: "Data EU only"
├─ Azure Policy: "MFA required"
├─ Azure Policy: "Encryption mandatory"
└─ Automatic enforcement
```
