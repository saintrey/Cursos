# 08. Authentication and Authorization (Autenticación y Autorización)

## Resumen del Capítulo
Este capítulo cubre fundamentales de seguridad de identidad en Azure. Explora la diferencia entre autenticación y autorización, Azure Active Directory (AAD), Zero Trust security, y múltiples métodos de validación de identidad.

## Temas Principales

### 1. Identity Services - Conceptos Fundamentales

#### a) Definición de Identity
- **Identity**: Quién ERES (como usuario/proceso)
- Azure necesita saber quién eres para dar acceso
- Foundation de todo en Azure security

#### b) Autenticación vs Autorización

**AUTENTICACIÓN (Authentication)**
```
¿Eres realmente quien dices ser?
→ Confirmar identidad
→ Username + password (o face, biometric, etc)
→ "Sí, eres realmente TÚ"
→ Primer test
```

Ejemplo: Credenciales correctas = Autenticado

**AUTORIZACIÓN (Authorization)**
```
Ahora que sais quién eres...
¿Tienes PERMISO para acceder a esto?
→ Granular permissions
→ "¿Puedes acceder a estos resources?"
→ Después de authentication
```

Ejemplo: Configuración de qué hace cada usuario

#### c) Orden Crítico
```
1. PRIMERO → Autenticación
   "¿Eres realmente tú?"
   
2. DESPUÉS → Autorización  
   "¿Qué puedes hacer?"

Sin autenticación → No hay autorización
```

#### d) Analogía Comida Rápida
```
Authentication:
  - Yo: "Quiero chips"
  - Sistema chequea mi identidad
  - "Eres realmente [nombre], welcome"

Authorization:
  - Sistema chequea mi "perfil"
  - Si diabético → "No chips para ti"
  - Si VIP → "Extra toppings gratis"
```

### 2. Azure Active Directory (AAD)

#### a) Definición General
- **Main tool** para administrar usuarios + permisos en Azure
- Pero: NO es lo mismo que "Active Directory" on-premises
- **Microsoft naming problem**: Similar names, different products
- Muy diferente skillset en AAD vs AD on-premises

#### b) Concepto Crítico
```
❌ NO PUEDES tener Azure account sin AAD
✅ Cada Azure account INCLUYE AAD
❌ AAD ≠ Active Directory (on-premises) - COMPLETAMENTE DIFERENTE
```

#### c) Componentes clave de AAD

**1. Tenant**
- Representación de tu organización en Azure
- Instancia dedicada de Azure AD
- PRIMER thing creado cuando signup a Azure
- Cada tenant es distinto y separado
- Un usuario puede ser: Member en 1 + Guest en 500 tenants máximo

**2. User**
- Persona o proceso
- Account en AAD
- Member o Guest

**3. Subscription**
- Entidad de billing
- Recibe invoice mensual
- Múltiples subscriptions en 1 tenant
- Si no pagas una subscription → Resources en esa subscription se apagan

#### d) Jerarquía AAD

```
TENANT (Tu organización)
├─ SUBSCRIPTION 1 (Billing entity)
│  └─ Resource Group
│     └─ Resources
└─ SUBSCRIPTION 2 (Billing entity)
   └─ Resource Group
      └─ Resources

Usuarios y permisos administrados por Tenant
Billing manejado por Subscriptions
```

#### e) Hybrid Cloud con AAD
- AAD puede amministrar usuarios ON-PREMISES + EN CLOUD
- Mismo usuario puede acceder ambos
- Hace hybrid cloud más manageable
- (Simple mention, details fuera de scope AZ-900)

### 3. Zero Trust Security Model

#### a) Problema: Trust Perimeter Viejo
```
Viejo Modelo (Pre-2020):
- Todo dentro red corporativa = TRUSTED
- Todo fuera = UNTRUSTED

Problema:
- Trabajo remoto es ahora normal
- Vieja perimeter desaparece
- Empleados acceden desde casa, coffee shop, etc
```

#### b) Solución: Zero Trust
```
Zero Trust Filosofía:
- NO asumir nada es trusted
- VERIFICAR SIEMPRE
- LEAST privilege (mínimos permisos necesarios)
```

#### c) Core de Zero Trust
- **"Never Trust, Always Verify"**
- Incluso usuarios internos deben re-verifiarse
- Cada acceso = Nueva validación
- Basado en identidad, no ubicación
- Habilita work-from-anywhere

#### d) Comparación

| Aspecto | Perimeter Model | Zero Trust |
|--------|-----------------|-----------|
| **Trust Location** | Dentro red corporativa | Basada en identidad |
| **Assumption** | Inside = good, Outside = bad | Siempre verify |
| **Remote Work** | Difícil | Perfect fit |
| **Scalability** | Limitada | Enterprise ready |

### 4. Multi-Factor Authentication (MFA)

#### a) Definición
- 2+ métodos de autenticación necesarios
- NO confiar en password solo

#### b) Ejemplo
```
1. Username + Password (factor 1)
2. SMS code to phone (factor 2)
3. → AMBOS necesarios = Autenticado

vs.

Solo password → Muy inseguro ahora
```

#### c) Tipos de Factores
- **Algo que sabes**: Password, PIN
- **Algo que tienes**: Teléfono, security key
- **Algo que eres**: Fingerprint, facial recognition

#### d) Beneficio
- Si password compromised, attacker still necesita 2do factor
- Muy difícil de breach

### 5. Conditional Access

#### a) Definición
- **Condiciones aplicadas a logins**
- Si condición no cumplida → Denegar acceso
- Protección beyond username/password

#### b) Ejemplos de Condiciones
```
Condición: "Usuario accede desde país sospechoso"
  → Denegar acceso O require adicional verification

Condición: "Device no tiene antivirus"
  → Denegar acceso
  
Condición: "IP no es corporativa"
  → Require MFA adicional
```

### 6. Passwordless Authentication

#### a) El Problema con Passwords
- Difíciles de recordar
- Fácil de guess
- Re-used en múltiples sitios
- Security nightmare

#### b) Solución: Passwordless
- Usa métodos más seguros
- Ejemplos: Facial recognition, phone approval, security keys
- Futuro de authentication

#### c) Beneficios
- Más seguro
- Mejor user experience
- Menos password resets

### 7. External Guest Access

#### a) Caso de Uso
```
Necesito dar acceso a usuarios EXTERNOS
(consultants, partners, clientes)
Sin crear nueva cuenta Azure completamente
```

#### b) Solución: Guest Access
- Invita usuarios externos
- Usan SUS PROPIOS ACCOUNTS (Gmail, LinkedIn, etc)
- Acceden tus resources bajo scope limitado
- Empresa no crea/maneja sus accounts

#### c) Beneficios
- Flexibilidad
- Guest vs Member
- Collaborate sin complex onboarding

### 8. Azure Active Directory Domain Services

#### a) Tradición: LDAP + Kerberos
- Viejo standard para domain authentication
- "Classic Active Directory" features
- Aplicaciones legacy usan esto

#### b) Azure AD Domain Services
- Brinda classic AD features en Azure
- Fully managed (no need to maintain)
- No necesitas on-premises domain controllers
- Compatible con legacy apps

#### c) Casos de Uso
- Legacy applications que NECESITAN classic AD
- LDAP authentication
- Kerberos constrained delegation

### 9. Single Sign-On (SSO)

#### a) Problema
```
Usuario necesita login a 5 aplicaciones
5 passwords diferentes
5 logins
Very tedious
```

#### b) Solución: SSO
```
1 login a AAD
   ↓
Acceso automático a TODOS los apps integrados
```

#### c) Beneficio
- User experience mucho mejor
- Menos password manage
- Enterprise standard

#### d) Integración Azure AD
- Integra con múltiples apps
- Microsoft apps (natural fit)
- Third-party apps (Slack, Salesforce, etc)

### 10. Resumen: Componentes de Identity

```
AZURE ACTIVE DIRECTORY (Centro)
├─ Auenticación (¿Eres realmente tú?)
│  ├─ Username/password
│  ├─ MFA (2+ factors)
│  └─ Passwordless (facial, phone)
│
├─ Autorización (¿Qué puedes hacer?)
│  ├─ RBAC (Role-Based Access Control)
│  └─ Conditional Access (reglas adicionales)
│
├─ Usuarios
│  ├─ Members
│  └─ Guests (external)
│
└─ Integración
   ├─ SSO (múltiples apps)
   ├─ Zero Trust (verify always)
   └─ Legacy AD (domain services)
```


## Jerarquía de Decisión

```
Usuario quiere acceder
    ↓
¿Eres quién dices ser?    → AUTENTICACIÓN
    ↓ SÍ
¿Tienes permiso?          → AUTORIZACIÓN
    ↓ SÍ
Acceso GRANT
    ↓ NO
Acceso DENY
```
